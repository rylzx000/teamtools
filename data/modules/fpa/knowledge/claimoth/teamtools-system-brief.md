# 非车险理赔核心系统 FPA系统知识简包

系统编码：`claimoth`  
系统名称：`非车险理赔核心系统`  
资料用途：供 TeamTools 后续进行 FPA 工作量评估时拼入请求包，辅助理解系统边界、业务链路、数据责任和复用判断。本文只保留系统级判断背景，不代表任何具体需求的评估结论。

## 1. 系统定位

【明确】非车险理赔核心系统属于非车险理赔主链路核心系统，承担核心交易、流程任务、状态流转和赔付处理职责。资料显示其覆盖报案、预报案、调度、查勘、案件处理、立案、定损、理算、核赔、结案、重开、支付信息修改、打印、第三方机构、注销拒赔、特殊任务、批量处理、未决跟踪和风控提示等环节。

【推断】本系统是非车险理赔业务的主处理边界，不只是查询或外围服务平台。涉及赔付金额、核赔审批、支付、结案、批量案件、风控规则或跨环节状态流转的需求，应按核心链路影响判断，不宜仅按页面字段或接口透传理解。

【待确认】现有资料主要来自 2023 年操作手册结构化页、表结构初步提要和文件树初步提要，不直接等同当前生产实现。若本次需求涉及当前运行边界、接口方向或表归属，应结合代码、接口文档或业务确认补证。

| 承保/保单数据 | 【明确】报案、立案、定损、核赔、打印多处引用保单、标的、限额、条款、客户信息 | 【推断】承保系统维护保单主数据 | 通常作为外部引用数据候选；若只是理赔侧快照或展示字段，需另判 |

| 支付与收付 | 【明确】结案依赖支付成功，支付信息修改面向未付款登记或支付失败计算书 | 【待确认】支付执行、回执、付款登记的系统归属和接口方向 | 账户修改申请和审核多为本系统输入过程；支付结果回写需确认 |

| 第三方机构 | 【明确】存在第三方机构申请、审核、评价、排行榜和领域信息查看 | 【待确认】机构名录、人员资质、排行榜来源和维护归属 | 申请、审核、评价更像本系统过程；机构主数据可能为外部引用或本系统维护 |

## 4. 关键业务链路

1. 【明确】报案/预报案链路：用户登记出险信息，可能暂存、提交、作废或由预报案转正式报案。本链路维护报案或预报案数据，并可能引用保单、客户、历史赔付、风险提示等外部或既有数据。FPA 评估时应重点区分登记/修改输入、查看查询、摘要或材料输出。

2. 【明确】调度与查勘链路：报案后进入任务处理，调度环节指派查勘机构、人员和地址，查勘环节录入事故采集、查勘要素、损失或事故者信息，并可能生成或发送查勘报告。若需求只调整地图、视频查勘入口或展示字段，需判断是否只是外部能力调用。

3. 【明确】立案、定损、理算、核赔链路：案件经查勘后进入立案、损失认定、赔付计算和核赔审核。该链路涉及案件状态、损失对象、赔付方案、领款人、计算书、核赔意见等核心数据。涉及赔付金额、费用、计算书、审批退回或提交上级的改造，通常是核心影响点。

4. 【明确】支付、结案和后续处理链路：核赔通过后进入支付相关处理，支付成功且无未完成插件流程时结案。支付信息修改支持对未付款登记或支付失败计算书调整账户信息并审核；结案后可能存在重开、归档、通赔评价等后续任务。支付执行归属、支付回执和归档任务是否独立需确认。

5. 【明确】第三方机构链路：业务可发起聘请第三方机构，选择机构和人员，提交申请、审核并评价；页面还可查看排行榜和领域信息。申请、审核、评价更偏本系统输入过程，机构名录、人员资质和排行榜数据维护责任仍需确认。

6. 【明确】健康险批量和通知链路：健康险大批量处理支持批次文件导入、一键核赔、定时任务执行和企业微信批次结果通知。导入、批次处理、失败清单、统计结果和通知触发可能影响不同事务候选；文件列数、重复校验和数据量上限不单独拆分。

7. 【明确/待确认】未决跟踪与风控链路：资料显示未决跟踪展示估损金额并存在提交确认、企业微信提醒；风控规则需区分风控平台规则与其他系统规则，理算/核赔提交只校验部分规则。规则来源、结果是否落库、提醒触发条件和通知对象需结合需求确认。

## 7. FPA判断辅助

以下内容只是系统级候选背景，不代表某个具体需求一定计入功能点。正式评估必须结合本次需求范围判断。

### 7.1 ILF 候选

- 【推断】报案、预报案、赔案/立案、查勘、定损、事故者/医疗费用、理算结果、计算书、领款人、支付账户修改、重开、注销拒赔、第三方机构申请评价、健康险批次、未决跟踪等，是本系统维护数据候选。
- 【待确认】流程状态、核赔意见、结案归档、风控结果、打印记录是否独立形成业务可识别数据组，需要结合需求和表结构确认。
- 【明确】日志、临时、备份、流程引擎、页面配置、导入暂存、DTO 和模板列默认不单独作为业务数据组。

### 7.2 EIF 候选

- 【推断】承保保单、客户、条款、标的、限额、银行/开户行、外部风控规则、第三方机构名录、影像本体、地图/地块数据等，在外部系统维护且被本系统稳定引用时，可作为外部引用数据候选。
- 【待确认】若上述数据只是接口实时返回字段、页面透传字段或理赔侧快照，不应直接按外部引用数据处理。

### 7.3 EI 候选

- 【明确】新增或修改报案、预报案、调度、查勘、发起任务、资料收集、立案、立案审核、定损维护、定损导入、发起理算、理算提交、领款人登记、核赔提交、结案、重开申请/审核、支付信息修改、第三方机构申请/审核/评价、注销拒赔申请/审核、健康险批次导入、未决提交确认等，通常是输入类候选。
- 【明确】审核通过、退回、提交上级、收回、作废、转正式报案、改派等改变业务状态的动作，应按业务过程判断，不按按钮数量判断。

### 7.4 EO 候选

- 【明确】查勘报告生成/发送、定损清单下载、失败数据下载、领款人信息下载、赔偿确认书发送、打印管理、保单抄件下载/打印、健康险批次结果企业微信通知、未决提醒等，是输出类候选。
- 【推断】分摊试算、复杂汇总、赔付计算结果、批处理统计结果，若形成用户可识别派生输出，也可能按输出类判断。

### 7.5 EQ 候选

- 【明确】任务查询、报案查看、保单信息查看、限额查询、开户行查询、查看计算书、定损关联记录查看、特殊任务查询、第三方机构排行榜/领域信息查看、未决跟踪展示、风控规则展示等，通常是查询类候选。
- 【明确】查询若只展示已有数据且不维护状态、不生成文件、不发送通知、不产生复杂派生结果，优先按查询类判断。

## 8. 复用程度判断口径

| 可复用对象 | 适用场景 | 倾向复用程度 | 降低复用的情况 |
|---|---|---|---|
| 既有页面字段和查询条件 | 新增展示字段、必录校验、提示文案、列表列、简单筛选 | 高 | 跨系统查询、复杂汇总、影响保存规则或权限路径 |
| 报案/预报案/查勘/立案等既有输入流程 | 在既有页面保存字段、调整提交校验或状态提示 | 高到中 | 新增独立业务对象、转化规则、审批流或跨环节联动 |
| 定损、理算、核赔核心流程 | 调整损失录入、赔付计算、核赔规则、退回路径 | 中 | 影响赔付金额、计算书、审批层级、风控校验或支付前置 |
| 导入导出和失败清单机制 | 模板列调整、失败数据下载、导入校验调整 | 高到中 | 新增异步批处理、批次数据、统计通知或大批量机制 |
| 企业微信和对外通知 | 调整已有通知内容或文案 | 高到中 | 新增触发条件、收件人规则、失败重试、独立任务或统计输出 |
| 健康险批量处理 | 在既有批次、一键核赔和通知链路上调整 | 中 | 新增批次模型、外部交互、定时处理规则或结果数据保存 |
| 第三方机构流程 | 调整申请、审核、评价字段或查看范围 | 中 | 新增机构名录维护、人员资质、排行榜算法或完整新流程 |
| 风控规则展示和校验 | 调整展示字段、规则来源过滤、提交校验 | 中 | 新增规则维护、跨平台规则合并、结果落库、通知或阻断策略 |
| 无正文目录或未知接口 | 追偿、诉讼、稽核、反欺诈调查、恢复、发票登记、监管、消息、综合查询等 | 不确定 | 当前资料不足，应先补证再判断 |

【明确】不能把“属于同一系统”直接等同于高复用。核心系统小改动可以高复用，但涉及主链路状态、赔付金额、理算核赔支付、结案、批量处理、风控规则、跨流程联动或外部系统交互时，应优先按中或低复用评估，再根据复用证据收敛。

## 9. 常见误判提醒

- 【明确】单个字段、按钮、提示文案、返回码、页面入口、查询条件、列表列、模板列、接口 DTO、权限显隐、静态资源、日志记录，通常不单独形成可识别功能。
- 【明确】必录、格式、重复、上限、风险弹窗、资料时间软提示等校验，通常并入对应提交或导入过程；只有形成独立规则维护、报告、通知或状态流转时才另行判断。
- 【明确】外部系统返回字段、风险标签、枚举值、提示语、接口透传字段，不直接构成外部引用数据；必须确认存在外部维护的稳定逻辑数据组。
- 【明确】查询只是展示已有数据时通常按查询类判断；若生成派生结果、文件、报表、通知、打印件、批处理统计或对外发送内容，可能转为输出类判断。
- 【明确】导入功能不按文件列、模板列、校验条数、失败提示数量重复拆分；导入过程、失败清单、批处理结果和通知是否拆分，要看是否为用户可识别过程。
- 【明确】不同险类页面展示差异不自动重复计算同一基本过程。只有新增险类专属流程、独立数据对象或独立输出时才考虑拆分。
- 【明确】`LOG`、`TEMP`、`BACKUP`、`ACT_*`、`WFL_*`、页面配置表、流程引擎表通常不作为业务数据功能；业务可识别的历史查询或归档维护需求除外。
- 【待确认】`PLC_*`、`PRD_*`、`PRT_*`、银行、风控、第三方机构等数据维护归属未完全确认，评估时不要直接强判为本系统维护。
- 【明确】历史候选清单和示例只能作为判断背景，不能整包直接计入，也不能把历史示例固化为系统永久规则。

## 10. 资料缺口与待确认点

| 问题 | 影响 | 建议确认方式 |
|---|---|---|
| 当前生产系统是否仍以 `newclaimoth` 和非车理赔数据库 Schema 为主要运行边界 | 影响系统边界和数据责任判断 | 向系统负责人确认运行架构、库表、服务边界 |
| `PRPLREGIST`、`PRPLCLAIM`、`PRPLCHECK`、`PRPLDAMAGE`、`PRPLPERSONINJURY`、`PRPLCOMPENSATE`、`PRPLPLAN`、`PRPLPAYEEINFO` 等候选表真实含义 | 影响本系统维护数据组判断 | 查询表注释、字段说明、代码引用或业务数据字典 |
| 保单、客户、条款、开户行、第三方机构、风控规则由哪个系统维护 | 影响外部引用数据与本系统数据的区分 | 确认接口文档、主数据归属和同步方式 |
| 理算、核赔、支付、结案之间实际状态流转和接口方向 | 影响核心链路拆分和复用判断 | 查看流程配置、接口调用、状态流转说明 |
| 健康险批量处理的批次表、定时任务、通知发送和失败数据保存方式 | 影响批处理输入、输出和数据责任判断 | 查看批处理设计、任务配置、批次结果表 |
| 影像管理是否只调用外部系统，还是本系统维护影像索引、资料收集状态或上传数量 | 避免把外部影像能力误判为本系统数据 | 确认影像接口、资料收集字段和上传记录归属 |
| 风控平台、百融、凯泰铭、医审平台、中银保信规则结果是否落库 | 影响查询、外部引用和本系统保存结果判断 | 查看风控接口、结果表、规则来源说明 |
| 第三方机构名录、排行榜、人员资质和评价由本系统还是外部系统维护 | 影响第三方机构数据组和查询输出判断 | 确认机构主数据来源、评价表、排行榜生成方式 |
| 追偿、诉讼、稽核、反欺诈调查、恢复、发票登记、监管、消息、综合查询等无正文目录是否仍有现行功能 | 影响资料覆盖完整性 | 补充现行操作手册、页面截图、接口或业务说明 |
| 当前资料时效和结构化手册覆盖范围 | 影响证据可信度 | 对关键需求选择性读取源码、接口、表字段和现行业务说明 |
