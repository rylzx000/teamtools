# 技术架构

## 目标

构建一个轻量、可扩展的内网 AI 工具平台。首版优先跑通 FPA 工作量评估模块，后续可扩展核心系统运维处理等模块。

当前 MVP 不把模型调用放在服务器侧，采用：

```text
后端生成 AI 请求包和记录任务状态
前端调用 DeepSeek 等外部模型
后端接收模型结果、校验 JSON 并生成 Excel
服务器使用 Docker Compose 部署
```

## 架构总览

```text
浏览器
  -> 前端页面
      -> 上传 Markdown / 文本
      -> 获取后端生成的 AI 请求包
      -> 调用外部大模型 API
      -> 接收 AI 结构化结果
      -> 接收模型调用错误
      -> 回传后端

后端 FastAPI
  -> 用户认证
  -> 任务管理
  -> 文件管理
  -> AI 请求包生成
  -> AI 调用结果接收
  -> AI 结果 JSON 校验
  -> Excel 脚本生成
  -> 结果下载
  -> 静态页面托管

数据与文件
  -> SQLite / 后续 PostgreSQL
  -> data/teamtools.db
  -> data/modules
  -> data/tasks
  -> logs
```

## 核心分层

- 前端层：负责页面交互、上传输入、获取 AI 请求包、执行模型调用、展示进度和回传结果。
- API 层：负责认证、任务提交、任务查询、文件管理、AI 请求包生成、模型结果接收、健康检查和静态文件托管。
- 任务处理层：负责 JSON 校验、质量检查、Excel 生成、结果落盘和状态流转。
- 数据层：保存用户、模块配置、任务记录、文件记录、结果记录等最小状态。
- 文件层：保存上传原文、AI 原始结果、结构化 JSON、Excel 输出和运行日志。

## 模型调用边界

首版模型调用在前端或用户端完成，后端不直接请求外部模型。

这样设计是为了适配当前网络条件：

- 服务器不依赖公网模型访问能力。
- 用户终端可通过页面调用外部模型。
- 后端不保存或暴露公共模型 API Key。
- 后端统一生成 AI 请求包，避免前端硬编码提示词、系统资料路径和模块规则。
- 后端接收模型成功结果或失败信息，并记录完整任务状态和排查线索。

API Key 首版建议由用户在页面临时填写，并保存在浏览器本地；不把团队公共 Key 写入前端代码。

已通过浏览器直连 DeepSeek API 的最小测试，确认前端浏览器到模型 API 的调用流程可连通。开发前仍需使用正式可用账号再验证一次成功响应格式；如果后续模型 API 不允许浏览器跨域调用，当前主链路需要改用企业代理入口或后端模型代理模式。

后续如果服务器具备稳定模型访问能力，可再增加后端模型代理或服务端模型调用模式。

## AI 请求包边界

AI 请求包由后端生成，前端只负责执行调用。

后端生成请求包时负责：

- 读取模块提示词模板和输出契约。
- 读取用户上传或粘贴的需求输入。
- 读取用户选择的系统资料目录或无资料模式配置。
- 合并页面参数、默认参数和模块规则。
- 生成可直接提交给外部模型的请求内容。
- 保存请求包摘要和版本信息，便于任务排查。

前端执行模型调用时负责：

- 从后端获取当前任务的 AI 请求包。
- 使用用户本地填写的模型 API Key 调用外部模型。
- 展示模型调用中的页面状态。
- 将模型原始响应、结构化 JSON 或调用错误回传后端。

前端不得自行拼接完整模块提示词，不直接读取服务器系统资料目录，也不决定 FPA 计算规则。

## 任务主流程

```text
1. 用户登录系统
2. 选择业务模块，例如 FPA
3. 上传或粘贴需求文本
4. 后端创建任务并保存输入
5. 后端生成 AI 请求包
6. 前端获取请求包并调用模型
7. 前端将 AI 结果或模型调用错误回传后端
8. 后端校验 JSON 和枚举值
9. 后端执行模块脚本生成结果文件
10. 用户查看任务状态并下载 Excel
```

## FPA 首版职责划分

AI 负责：

- 理解需求内容。
- 拆分功能点候选项。
- 输出结构化 JSON。
- 给出不确定事项。

后端负责：

- 校验 JSON 格式和字段。
- 校验枚举值。
- 做基础质量检查。
- 调用脚本生成 Excel。
- 保留模板公式和格式。
- 保存任务结果并提供下载。

AI 不负责：

- 生成 Excel。
- 计算公式。
- 计算人天汇总。
- 修改模板。
- 写入最终文件。

## 模块边界

- 公共能力：认证、用户、任务、文件、配置、日志、结果下载。
- 业务模块：按 `fpa`、`ops` 等模块独立扩展。
- 模块之间通过公共任务、文件和配置能力协作，不直接修改其他模块内部实现。
- 每个模块应独立维护自己的输入契约、输出契约、脚本和资源文件。

## 部署方式

服务器推荐使用 Docker Compose 部署，直接通过端口访问。

```text
浏览器
  -> http://服务器IP:端口
      -> teamtools 容器
          -> FastAPI
          -> 前端静态文件
          -> SQLite 文件 / 后续 PostgreSQL
          -> data 挂载目录
          -> logs 挂载目录
```

首版不要求 Nginx。

本地开发可继续直接运行前后端，方便调试；服务器部署使用 Docker Compose，减少环境差异和依赖污染。

## 数据库选择

MVP 优先使用 SQLite，降低部署复杂度。

适用条件：

- 首批团队规模约 10 人。
- 任务量不高。
- 主要任务是上传、生成和下载。
- 不需要复杂并发写入。

后续推广到部门或并发上升时，可切换 PostgreSQL。

Redis 首版不需要；如后续引入服务端模型调用、长任务队列或高并发任务调度，再评估引入。

## 目录原则

代码、配置、数据目录统一放在工程目录下，避免项目复制或部署时丢失运行依赖。

推荐：

```text
teamtools/
├─ backend/
├─ frontend/
├─ docs/
├─ data/
│  ├─ teamtools.db
│  ├─ modules/
│  └─ tasks/
├─ logs/
├─ Dockerfile
├─ docker-compose.yml
└─ .env.example
```

`data/` 和 `logs/` 在 Docker 部署时挂载到宿主机，容器重建不能丢失用户上传文件、任务结果和数据库文件。

## 设计原则

- 先跑通完整链路，再优化提示词和模型效果。
- AI 处理非确定性理解，脚本处理确定性生成。
- 前端可以调用模型，但后端统一生成请求包并记录状态。
- 后端不能信任模型结果，必须校验。
- 业务模块独立，公共能力复用。
- 本地开发保持轻量，服务器部署保持可复现。
- 不为首版引入不必要的队列、Redis、Nginx 或微服务拆分。
