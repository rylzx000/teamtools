# 数据与文件存储

## 目标

本文档描述平台首版的数据、文件和运行产物存储规则。

当前 MVP 使用：

```text
SQLite + 本地文件目录 + Docker 挂载
```

首版不引入对象存储、Redis、分布式文件系统或独立文件服务。

## 存储原则

- 代码、文档、脚本和模板放在工程目录。
- 运行数据统一放在工程根目录下的 `data/`。
- 运行日志统一放在工程根目录下的 `logs/`。
- `data/` 和 `logs/` 不进入版本管理。
- Docker 部署时必须把 `data/` 和 `logs/` 挂载到宿主机。
- 容器重建、镜像更新或服务重启不能丢失上传文件、任务结果和数据库。

## 数据库

MVP 使用 SQLite。

本地开发默认路径：

```text
D:\project\teamtools\data\teamtools.db
```

Linux 服务器路径：

```text
/data/teamtools/data/teamtools.db
```

Docker 容器内路径：

```text
/app/data/teamtools.db
```

说明：

- SQLite 文件必须位于挂载目录中。
- 不要把数据库文件放在容器内部临时文件系统中。
- 后续如推广到部门或并发增加，可切换 PostgreSQL。

## Docker 挂载

Docker Compose 推荐挂载：

```yaml
volumes:
  - ./data:/app/data
  - ./logs:/app/logs
```

挂载后：

- 上传文件保存到宿主机 `./data`。
- Excel 输出保存到宿主机 `./data`。
- SQLite 文件保存到宿主机 `./data/teamtools.db`。
- 日志保存到宿主机 `./logs`。

## data 目录结构

建议结构：

```text
data/
├─ teamtools.db
├─ modules/
│  └─ fpa/
│     ├─ profile/
│     ├─ templates/
│     └─ scripts/
├─ tasks/
│  └─ fpa/
│     └─ {task_id}/
│        ├─ input/
│        ├─ ai/
│        ├─ output/
│        └─ runtime/
├─ uploads/
├─ outputs/
└─ runtime/
```

说明：

- `modules/` 保存模块运行资源，例如模板、脚本、系统资料摘要等。
- `tasks/` 保存按任务隔离的输入、AI 结果、输出和日志。
- `uploads/` 可保存全局上传临时文件。
- `outputs/` 可保存全局导出结果。
- `runtime/` 可保存临时状态、缓存或运行锁文件。

如果任务产物已经放在 `tasks/{module}/{task_id}/` 下，`uploads/` 和 `outputs/` 可以只作为兼容或临时目录。

## FPA 任务目录

FPA 单个任务建议目录：

```text
data/tasks/fpa/{task_id}/
├─ input/
├─ ai/
├─ output/
└─ runtime/
```

### input

保存用户输入：

```text
input/
├─ input.md
├─ 原始上传文件.md
└─ task_params.json
```

说明：

- 用户上传的 Markdown 原文需要保存。
- 用户粘贴文本可保存为 `input.md`。
- 页面选择参数保存为 `task_params.json`。

### ai

保存 AI 请求包和模型结果：

```text
ai/
├─ AI请求包.json
├─ AI请求摘要.json
├─ AI原始响应.json
├─ AI调用错误.json
├─ AI分析.md
└─ AI结构化结果.json
```

说明：

- `AI请求包.json` 保存后端生成并提供给前端调用模型的完整请求内容。
- `AI请求摘要.json` 保存请求包摘要、模型建议和必要追踪信息，便于排查。
- `AI原始响应.json` 保存模型原始响应，便于排查。
- `AI调用错误.json` 保存前端回传的模型调用失败信息，仅失败时生成。
- `AI分析.md` 可选保存，取决于前端是否回传 Markdown 分析说明。
- `AI结构化结果.json` 是后端校验和生成 Excel 的主要输入。

### output

保存正式输出：

```text
output/
├─ FPA生成过程.json
└─ FPA工作量评估.xlsx
```

说明：

- `FPA生成过程.json` 由后端脚本生成，记录脚本处理摘要、质量检查和估算结果。
- `FPA工作量评估.xlsx` 是用户下载的最终结果。

### runtime

保存运行日志和临时文件：

```text
runtime/
├─ task.log
├─ *.tmp
└─ error_detail.txt
```

说明：

- 普通用户不直接查看完整运行日志。
- 管理员可以查看日志和错误详情。
- `.tmp` 文件不登记为正式产物。

## 文件写入规则

关键文件生成时先写临时文件，成功后再改名。

示例：

```text
AI结构化结果.json.tmp -> AI结构化结果.json
FPA生成过程.json.tmp -> FPA生成过程.json
FPA工作量评估.xlsx.tmp -> FPA工作量评估.xlsx
```

规则：

- 正式文件生成成功后才写入文件索引。
- 任务失败时可以保留临时文件供管理员排查。
- 用户下载只允许下载正式产物。

## 文件索引

数据库需要记录任务文件索引，避免页面直接扫描目录。

建议记录：

- 任务 ID。
- 文件角色。
- 文件名。
- 文件相对路径。
- 文件大小。
- MIME 类型。
- 创建时间。
- 是否普通用户可见。

文件路径建议保存相对路径，例如：

```text
tasks/fpa/{task_id}/output/FPA工作量评估.xlsx
```

不要在数据库中长期保存本机绝对路径，避免迁移服务器后路径失效。

## 文件角色

建议文件角色：

| 角色 | 说明 | 普通用户可见 |
|---|---|---|
| input_file | 用户上传原始文件 | 是 |
| ai_request_package | 后端生成的 AI 请求包 | 通过专用接口获取，不在文件列表展示 |
| ai_request_summary | AI 请求包摘要 | 否 |
| ai_raw_response | AI 原始响应 | 否 |
| ai_call_error | AI 调用错误 | 否 |
| ai_analysis | AI 分析说明 | 是 |
| ai_structured_json | AI 结构化 JSON | 管理员可见 |
| generation_summary | 生成过程 JSON | 管理员可见 |
| result_excel | Excel 结果文件 | 是 |
| task_log | 任务日志 | 否 |
| error_detail | 错误详情 | 否 |

## 清理策略

MVP 不自动删除任务和产物。

原因：

- 便于用户下载历史结果。
- 便于管理员排查模型输出和生成问题。
- 便于后续做任务审计。

后续如数据量增加，再增加管理员清理能力。

## 备份建议

需要备份：

```text
data/
logs/
```

最低限度必须备份：

```text
data/teamtools.db
data/tasks/
data/modules/
```

如果后续使用 PostgreSQL，则数据库按 PostgreSQL 方式备份，文件目录仍需单独备份。

## 边界

- 不把上传文件、模型响应或 Excel 输出提交到代码仓库。
- 不把用户模型 API Key 写入文件产物。
- 不在日志中记录密钥、Cookie、Token 或完整请求头。
- 模块脚本只能写当前任务目录。
- 普通用户只能访问自己的任务文件。
- 管理员可以查看全部任务文件和错误详情。
