# 任务处理与结果生成设计

本文档描述平台首版任务处理、AI 请求包生成、模型结果接收、JSON 校验和结果文件生成机制。首版不引入服务端模型调用或独立后台任务进程作为核心链路。

## 1. 职责

首版后端不负责调用外部大模型。模型调用由前端或用户端完成，但 AI 请求包由后端统一生成。后端接收模型成功结果或失败信息，并执行确定性处理。

后端任务处理职责：

- 接收用户提交的任务。
- 保存上传文件和任务输入。
- 读取模块提示词模板、系统资料配置和页面参数。
- 生成当前任务的 AI 请求包。
- 接收前端回传的 AI 输出 JSON。
- 接收前端回传的模型调用错误。
- 校验 JSON 结构、字段和枚举值。
- 执行模块脚本生成结果文件。
- 保存正式产物和过程产物。
- 更新任务状态、文件索引和任务事件。
- 处理失败、取消、重新运行和日志。

## 2. 首版运行模型

MVP 采用轻量同步处理模型：

```text
API 接收 AI 结果
  -> 校验 JSON
  -> 调用模块处理器
  -> 生成 Excel
  -> 写入任务结果
```

约定：

- 首版不引入 Redis、Celery 或消息队列。
- 首版不需要独立后台任务进程。
- Excel 生成属于短任务，由后端 API 触发执行。
- 如果后续任务耗时明显增加，再扩展为后台任务进程或任务队列。

## 3. 为什么首版不在服务端调模型

当前网络和资源条件决定首版不适合服务器侧模型调用：

- 服务器不一定能访问公网模型。
- 内网模型资源有限，暂不作为首版稳定依赖。
- 用户终端具备调用外部模型的条件。
- 服务端直接调用模型的网络、额度和稳定性暂不满足首版要求。

因此 MVP 采用：

```text
后端生成 AI 请求包
前端调用模型
后端接收结果或错误
后端确定性生成 Excel
```

服务端模型调用可作为后续可选扩展，不进入首版主链路。

## 4. 任务状态

任务状态用于页面展示、失败排查和管理员查看。

首版建议状态：

| 状态 | 含义 |
|---|---|
| 草稿 | 用户已进入模块，但尚未正式提交 |
| 等待AI调用 | 已创建任务并生成 AI 请求包，等待前端调用模型并回传结果 |
| 结果校验中 | 后端正在校验 AI JSON |
| 生成结果中 | 后端正在调用脚本生成结果文件 |
| 已完成 | 已生成正式结果文件 |
| 失败 | 校验、脚本或文件处理失败 |
| 已取消 | 用户或管理员取消任务 |

说明：

- 首版没有“AI分析中”状态，因为 AI 调用不在后端执行。
- 前端可以在本地展示模型调用进度。
- 后端记录请求包生成、请求包获取、AI 结果回传和 AI 调用失败等事件，便于排查。

## 5. FPA 任务主流程

```text
1. 用户选择 FPA 模块
2. 用户上传 Markdown 或粘贴需求文本
3. 后端保存输入并创建任务
4. 后端生成 AI 请求包，状态为 等待AI调用
5. 前端获取 AI 请求包并调用模型
6. 前端拿到 AI 输出 JSON，或捕获模型调用错误
7. 前端回传 AI 输出 JSON 或错误信息
8. 后端收到 AI 输出后进入 结果校验中
9. 后端校验 JSON schema、枚举值和必要字段
10. 后端进入 生成结果中
11. 后端调用 FPA Excel 生成脚本
12. 后端登记正式产物
13. 后端标记任务 已完成
14. 用户下载 Excel
```

## 6. 输入文件处理

用户上传的原始文件需要保存，方便后续追溯和重新运行。

建议目录：

```text
data/tasks/fpa/{task_id}/input/
```

保存内容：

- 用户上传的 Markdown 文件。
- 用户粘贴文本生成的 `input.md`。
- 任务参数快照。

首版只支持 Markdown 和文本，不处理 PDF、Word、图片等复杂格式。

## 7. AI 请求包与结果接收

AI 请求包由后端生成并保存到任务目录，便于复现和排查。

建议目录：

```text
data/tasks/fpa/{task_id}/ai/
```

建议文件：

```text
AI请求包.json
AI请求摘要.json
```

请求包内容应包含：

- 模型输入消息或提示词文本。
- 输出 JSON 契约说明。
- 用户选择的系统和估算参数。
- 需求输入正文或后端允许前端提交给模型的文本内容。
- 不包含服务器绝对路径、密钥、Cookie 或用户不可见的内部配置。

前端回传 AI 输出时，后端需要保存原始响应和解析后的 JSON。

建议目录：

```text
data/tasks/fpa/{task_id}/ai/
```

建议文件：

```text
AI原始响应.json
AI调用错误.json
AI分析.md
AI结构化结果.json
```

首版可以先只要求前端回传 JSON；如果前端同时保留 Markdown 分析说明，则保存为 `AI分析.md`。如果模型调用失败，前端需要回传脱敏后的错误信息，后端保存为 `AI调用错误.json` 并标记任务失败。

## 8. JSON 校验

后端必须校验 AI 输出，不允许直接信任模型结果。

校验内容：

- 顶层必须是 JSON object。
- `items` 必须是非空数组。
- 必填字段必须存在。
- `category` 只能是 `ILF`、`EIF`、`EI`、`EO`、`EQ`。
- `reuse` 只能是 `高`、`中`、`低`。
- `change_type` 只能是 `新增`、`修改`、`删除`。
- 字符串字段不能全部为空。

校验失败时：

- 状态改为 `失败`。
- 写入用户可见错误摘要。
- 写入管理员错误详情。
- 保留 AI 原始响应。
- 不生成 Excel。

## 9. 质量检查

质量检查用于提示风险，首版不一定阻断生成。

建议检查项：

- 功能点条数过少。
- 功能点条数异常过多。
- 存在暂不做范围的功能点。
- 配置项被直接识别为 `ILF`。
- 总提交项和子提交项重复计数。
- 前端改造口径下遗漏状态展示、返回补拍、缺项提示或确认弹窗。
- `remark` 缺少类型、复用或修改依据。

质量检查结果写入任务结果，供用户和管理员查看。

## 10. Excel 生成

FPA 模块通过确定性脚本生成 Excel。

处理规则：

- 使用固定 Excel 模板。
- 只写入允许写入的输入单元格。
- 保留模板公式和样式。
- 根据功能点条数动态保留或扩展明细行。
- 不让 AI 生成 Excel。
- 不让 AI 计算人天。

建议目录：

```text
data/tasks/fpa/{task_id}/output/
```

建议文件：

```text
FPA生成过程.json
FPA工作量评估.xlsx
```

## 11. 输出文件落地

关键文件生成时应先写临时文件，成功后再改名为正式文件。

示例：

```text
AI结构化结果.json.tmp -> AI结构化结果.json
FPA生成过程.json.tmp -> FPA生成过程.json
FPA工作量评估.xlsx.tmp -> FPA工作量评估.xlsx
```

规则：

- `.tmp` 文件不登记为正式产物。
- 正式文件生成后才写入文件索引。
- 如果任务失败，保留临时文件供管理员排查。
- 普通用户不能查看临时文件。

## 12. 成功处理

任务成功时：

- 状态改为 `已完成`。
- 写入 `finished_at`。
- 写入结果摘要。
- 写入正式文件索引。
- 写入 `task_completed` 事件。
- 保留上传文件、AI 原始结果、结构化 JSON、生成过程和 Excel。

FPA 成功任务建议写入：

- 功能点条数。
- 调整后工作量中值。
- 质量检查结果。
- Excel 结果文件路径。
- 生成过程 JSON 路径。

## 13. 失败处理

任一阶段失败时：

- 状态改为 `失败`。
- 写入 `failure_stage`。
- 写入普通用户可见的 `error_summary`。
- 写入管理员可见的 `admin_error_detail`。
- 写入 `finished_at`。
- 写入 `task_failed` 事件。
- 保留输入文件、AI 原始结果、日志和临时文件。
- 不自动重跑。

用户或管理员可以通过“重新运行”新建任务。原失败任务不覆盖。

## 14. 取消处理

首版取消只作用于后端任务状态。

处理规则：

- 如果任务仍处于 `等待AI调用`，可以取消。
- 如果任务正在 `结果校验中` 或 `生成结果中`，尽力取消。
- 如果任务已完成、失败或取消，不再执行取消。
- 取消任务不生成正式结果文件。

如果前端正在调用模型，前端需要自行中止请求或提示用户关闭等待。

## 15. 重新运行

重新运行处理规则：

- 不覆盖原任务。
- 新建任务 ID。
- 复制原任务输入文件和任务参数。
- 重新生成 AI 请求包，并等待前端调用模型和回传结果。
- 原任务保留。
- 通过 `rerun_from_task_id` 关联原任务。

重新运行适用于：

- JSON 校验失败。
- 用户修改需求后重新生成。
- 用户认为 AI 结果质量不满意。
- Excel 生成失败后修复配置再运行。

## 16. 日志

每个任务保存一份处理日志。

建议路径：

```text
data/tasks/fpa/{task_id}/runtime/task.log
```

约定：

- 普通用户只看错误摘要和任务事件。
- 管理员可以查看完整日志。
- 日志不输出模型 API Key、密钥、Cookie 或敏感环境变量。
- 数据库只保存摘要，完整日志保存在文件系统。

## 17. 任务事件

后端需要将关键动作写入任务事件。

建议记录：

- 任务创建。
- 文件上传完成。
- AI 请求包生成完成。
- 前端获取 AI 请求包。
- AI 调用失败。
- AI 结果回传。
- JSON 校验开始。
- JSON 校验失败。
- JSON 校验成功。
- Excel 生成开始。
- Excel 生成完成。
- 用户请求取消。
- 任务失败。
- 任务完成。
- 重新运行创建。

事件用于：

- 任务详情展示。
- 管理员排查。
- 后续审计。

## 18. 模块分发

任务处理层按 `module` 分发到不同模块处理器。

示例：

```text
module = fpa -> FPA 结果生成处理器
module = ops -> OPS 处理器
```

首版只实现 FPA 处理器，但结构上预留 OPS 和其他后续模块。

## 19. FPA 阶段流转

FPA 首版阶段：

| 阶段 | 状态 |
|---|---|
| 创建任务并生成请求包 | 等待AI调用 |
| 前端回传 AI 结果 | 结果校验中 |
| JSON 校验通过 | 生成结果中 |
| Excel 生成完成 | 已完成 |
| 任意阶段失败 | 失败 |
| 用户取消 | 已取消 |

每次阶段切换都需要更新任务状态，并写入任务事件。

## 20. 部署关系

Docker Compose 部署下，首版可以只运行一个应用容器。

```text
teamtools 容器
  -> FastAPI API
  -> 前端静态文件
  -> JSON 校验
  -> Excel 生成脚本
  -> data 挂载目录
  -> logs 挂载目录
```

首版不需要独立后台任务容器。

以下仅作为非 MVP 扩展示意，不属于首版部署范围。只有在后续引入服务端模型调用、长任务队列、高并发调度或部门级数据库能力时，才重新评估：

```text
teamtools-api 容器
teamtools-worker 容器
postgres 容器
redis 容器（可选）
```

首版开发和部署不得按上面的扩展结构实现，避免误引入 Worker、Redis 或 PostgreSQL。

## 21. 权限边界

任务处理必须遵守以下边界：

- 不修改源码目录。
- 不读取无关任务目录。
- 不读取模型 key、环境变量和系统敏感路径。
- 普通用户只能访问自己的任务和产物。
- 管理员可以查看全部任务、错误详情和完整日志。
- 生成脚本只能写入当前任务目录。

## 22. 后续扩展

后续可扩展方向：

- 服务端模型代理。
- 后端直接调用内网模型。
- 独立后台任务进程。
- 多任务并发。
- Redis 或消息队列。
- PostgreSQL。
- 管理员任务统计和监控。
- 更细的质量检查规则。

这些能力首版不实现。
