# 数据库设计

## 1. 设计目标

平台首版使用 SQLite，后续如果推广到部门，可迁移到 PostgreSQL。

数据库需要满足：

- 支持用户登录和管理员查看全部任务。
- 支持多模块扩展，首个业务模块为 FPA。
- 支持后端生成 AI 请求包，前端执行模型调用后回传 AI 结构化结果或调用错误。
- 支持后端校验 JSON、生成 Excel 和记录结果。
- 支持任务文件、任务事件、失败排查和重新运行。
- 避免把某个模块的专属字段塞进全局任务表。

总体采用：

```text
统一任务表 + 统一文件表 + 统一事件表 + 各模块扩展表
```

MVP 核心表：

| 表名 | 中文说明 |
|---|---|
| `users` | 用户与权限表 |
| `tasks` | 统一任务主表 |
| `task_files` | 统一任务文件索引表 |
| `task_events` | 统一任务事件表 |
| `fpa_task_details` | FPA 任务扩展表 |

首版暂不单独设计 `system_profiles` 表，系统资料和模块资源先放在 `data/modules/` 文件目录下。后续如果需要页面维护系统资料，再新增系统资料表。

## 2. 通用约定

- 主键统一使用字符串 ID，便于跨表关联、文件目录命名和日志排查。
- 时间字段统一保存带时区的 ISO 时间字符串，便于 SQLite 使用。
- JSON 类字段首版使用 `TEXT` 保存 JSON 字符串，后续迁移 PostgreSQL 时可改为 `JSONB`。
- 文件路径统一保存相对 `TEAMTOOLS_DATA_DIR` 的路径，不保存服务器绝对路径。
- 普通用户权限通过 `tasks.created_by` 判断。
- 管理员权限通过 `users.role = admin` 判断。
- 重新运行任务不覆盖原任务，通过 `tasks.rerun_from_task_id` 关联。
- 完整日志和大体积排查文件保存到文件系统，数据库只保存摘要、路径和必要索引。

## 3. 用户表 users

`users` 保存平台用户和最小权限信息。首版不设计组织架构、部门树或复杂权限模型。

| 字段 | 类型 | 必填 | 中文备注 |
|---|---|---|---|
| `id` | TEXT | 是 | 用户 ID |
| `username` | TEXT | 是 | 登录账号，唯一 |
| `display_name` | TEXT | 是 | 页面展示名称 |
| `password_hash` | TEXT | 是 | 密码哈希，不保存明文密码 |
| `role` | TEXT | 是 | 用户角色，普通用户为 `user`，管理员为 `admin` |
| `enabled` | INTEGER | 是 | 是否启用，`1` 启用，`0` 禁用 |
| `created_at` | TEXT | 是 | 创建时间 |
| `updated_at` | TEXT | 是 | 更新时间 |
| `last_login_at` | TEXT | 否 | 最近登录时间 |

建议约束：

- `username` 唯一。
- `role` 首版只允许 `user`、`admin`。

## 4. 统一任务主表 tasks

`tasks` 只保存所有模块通用的任务字段。模块专属字段放到各自扩展表。

### 4.1 字段设计

| 字段 | 类型 | 必填 | 中文备注 |
|---|---|---|---|
| `id` | TEXT | 是 | 任务 ID |
| `module` | TEXT | 是 | 模块编码，例如 `fpa`、`ops` |
| `title` | TEXT | 是 | 任务标题或需求名称 |
| `status` | TEXT | 是 | 任务状态 |
| `created_by` | TEXT | 是 | 提交用户 ID，关联 `users.id` |
| `created_at` | TEXT | 是 | 创建时间 |
| `updated_at` | TEXT | 是 | 最近更新时间 |
| `submitted_at` | TEXT | 否 | 用户正式提交任务时间 |
| `ai_request_created_at` | TEXT | 否 | 后端生成 AI 请求包时间 |
| `ai_request_fetched_at` | TEXT | 否 | 前端最近一次获取 AI 请求包时间 |
| `ai_result_received_at` | TEXT | 否 | 前端回传 AI 结果或调用错误时间 |
| `processing_started_at` | TEXT | 否 | 后端开始校验或生成结果时间 |
| `finished_at` | TEXT | 否 | 任务完成、失败或取消时间 |
| `cancel_requested` | INTEGER | 是 | 是否请求取消，`1` 是，`0` 否 |
| `cancelled_at` | TEXT | 否 | 取消完成时间 |
| `cancelled_by` | TEXT | 否 | 取消人用户 ID，关联 `users.id` |
| `failure_stage` | TEXT | 否 | 失败阶段，例如 `json_validation`、`excel_generation` |
| `error_summary` | TEXT | 否 | 普通用户可见的脱敏错误摘要 |
| `admin_error_detail` | TEXT | 否 | 管理员可见的错误详情摘要，不保存超大完整日志 |
| `rerun_from_task_id` | TEXT | 否 | 重新运行来源任务 ID，关联 `tasks.id` |

### 4.2 任务状态

首版任务状态：

| 状态 | 中文说明 |
|---|---|
| `draft` | 草稿，用户已进入模块但尚未正式提交 |
| `waiting_ai_call` | 等待 AI 调用，任务已创建且请求包已生成，等待前端调用模型并回传结果 |
| `validating_result` | 结果校验中，后端正在校验 AI JSON |
| `generating_result` | 生成结果中，后端正在调用脚本生成 Excel |
| `completed` | 已完成，正式结果文件已生成 |
| `failed` | 失败，校验、脚本或文件处理失败 |
| `cancelled` | 已取消，用户或管理员取消任务 |

说明：

- 首版没有 `ai_analyzing` 状态，因为 AI 调用不在后端执行。
- 如果前端需要展示模型调用中状态，由前端本地控制。
- 后端记录 AI 请求包生成、请求包获取、AI 结果回传和 AI 调用失败事件。

### 4.3 建议索引

| 索引字段 | 中文说明 |
|---|---|
| `module, status, created_at` | 按模块和状态查询任务 |
| `created_by, module, created_at` | 普通用户查询自己的模块任务 |
| `rerun_from_task_id` | 查询重新运行关系 |
| `created_at` | 管理员按时间查看任务 |

## 5. FPA 任务扩展表 fpa_task_details

`fpa_task_details` 保存 FPA 模块专属字段。每个 FPA 任务对应一条扩展记录。

| 字段 | 类型 | 必填 | 中文备注 |
|---|---|---|---|
| `task_id` | TEXT | 是 | 任务 ID，关联 `tasks.id`，同时作为主键 |
| `system_code` | TEXT | 是 | 用户选择的系统编码 |
| `system_name` | TEXT | 是 | 提交时的系统名称快照 |
| `assessment_mode` | TEXT | 是 | 估算口径，例如 `frontend_change`、`backend_change`、`system_function` |
| `count_timing` | TEXT | 是 | 规模计数时机，默认 `估算早期` |
| `no_knowledge_mode` | INTEGER | 是 | 是否无资料模式，`1` 是，`0` 否 |
| `target_person_days` | REAL | 否 | 用户填写的目标人天 |
| `ai_item_count` | INTEGER | 否 | AI 输出功能点条数 |
| `result_item_count` | INTEGER | 否 | 实际写入 Excel 的功能点条数 |
| `result_median_person_days` | REAL | 否 | 后端脚本按模板公式口径计算的结果中值 |
| `target_hit` | INTEGER | 否 | 是否命中目标人天，`1` 是，`0` 否，空表示未计算 |
| `quality_flags` | TEXT | 否 | 质量提示标记，JSON 字符串 |
| `validation_errors` | TEXT | 否 | JSON 校验错误，JSON 字符串 |

说明：

- `assessment_mode` 首版可以先使用默认值，页面能力后续再细化。
- `ai_item_count` 记录模型原始输出条数。
- `result_item_count` 记录最终进入 Excel 的条数；如果首版不做人为修正，两者通常一致。
- `quality_flags` 不一定阻断生成，主要用于提示风险。

建议索引：

| 索引字段 | 中文说明 |
|---|---|
| `system_code` | 按系统筛选或统计 FPA 任务 |
| `assessment_mode` | 按估算口径筛选 |
| `target_hit` | 查询目标命中情况 |
| `no_knowledge_mode` | 排查无资料模式任务 |

## 6. 统一任务文件表 task_files

`task_files` 保存所有模块的任务文件索引。文件实际内容保存在运行数据目录。

| 字段 | 类型 | 必填 | 中文备注 |
|---|---|---|---|
| `id` | TEXT | 是 | 文件记录 ID |
| `task_id` | TEXT | 是 | 任务 ID，关联 `tasks.id` |
| `file_role` | TEXT | 是 | 文件角色，例如 `result_excel` |
| `storage_path` | TEXT | 是 | 文件相对路径，相对 `TEAMTOOLS_DATA_DIR` |
| `original_name` | TEXT | 否 | 用户上传时的原始文件名 |
| `display_name` | TEXT | 否 | 页面展示文件名 |
| `mime_type` | TEXT | 否 | 文件 MIME 类型 |
| `size_bytes` | INTEGER | 否 | 文件大小，单位字节 |
| `sha256` | TEXT | 否 | 文件 SHA-256 摘要，用于排查和完整性校验 |
| `downloadable` | INTEGER | 是 | 是否允许下载，`1` 是，`0` 否 |
| `viewable` | INTEGER | 是 | 是否允许页面查看，`1` 是，`0` 否 |
| `admin_only` | INTEGER | 是 | 是否仅管理员可见，`1` 是，`0` 否 |
| `created_at` | TEXT | 是 | 文件记录创建时间 |

### 6.1 FPA 首版文件角色

| file_role | 中文说明 | 普通用户权限 |
|---|---|---|
| `input_pasted` | 粘贴文本保存文件 | 页面查看、复制、重新运行 |
| `input_uploaded` | 上传 Markdown 文件 | 页面查看、复制、重新运行 |
| `task_params` | 任务参数快照 | 页面查看 |
| `ai_request_package` | AI 请求包 | 普通用户通过专用接口获取，不在普通文件列表展示、不提供下载；管理员可查看排查文件 |
| `ai_request_summary` | AI 请求包摘要 | 管理员查看 |
| `ai_raw_response` | AI 原始响应 | 管理员查看 |
| `ai_call_error` | AI 调用错误 | 管理员查看 |
| `ai_analysis` | AI 分析说明 | 管理员查看、复核 |
| `ai_structured_json` | AI 结构化 JSON | 管理员查看 |
| `generation_summary` | 生成过程 JSON | 管理员查看 |
| `result_excel` | Excel 结果文件 | 下载 |
| `task_log` | 任务日志 | 管理员查看 |
| `error_detail` | 错误详情 | 管理员查看 |

### 6.2 建议索引

| 索引字段 | 中文说明 |
|---|---|
| `task_id, file_role` | 按任务和文件角色查找产物 |
| `sha256` | 文件完整性排查 |

## 7. 统一任务事件表 task_events

`task_events` 保存任务状态变化和关键操作日志。事件用于页面展示、排查和审计。

| 字段 | 类型 | 必填 | 中文备注 |
|---|---|---|---|
| `id` | TEXT | 是 | 事件 ID |
| `task_id` | TEXT | 是 | 任务 ID，关联 `tasks.id` |
| `event_type` | TEXT | 是 | 事件类型 |
| `message` | TEXT | 是 | 事件摘要，普通用户可见时需要脱敏 |
| `detail_json` | TEXT | 否 | 事件详情，JSON 字符串 |
| `created_by` | TEXT | 否 | 触发事件的用户 ID，系统事件可为空 |
| `created_at` | TEXT | 是 | 事件创建时间 |

常见事件类型：

| event_type | 中文说明 |
|---|---|
| `task_created` | 任务创建 |
| `file_uploaded` | 用户文件上传完成 |
| `ai_request_created` | 后端生成 AI 请求包 |
| `ai_request_fetched` | 前端获取 AI 请求包 |
| `ai_call_failed` | 前端回传模型调用失败 |
| `ai_result_received` | 前端回传 AI 结果 |
| `json_validation_started` | JSON 校验开始 |
| `json_validation_failed` | JSON 校验失败 |
| `json_validation_passed` | JSON 校验通过 |
| `excel_generation_started` | Excel 生成开始 |
| `excel_generated` | Excel 生成完成 |
| `cancel_requested` | 用户请求取消 |
| `task_cancelled` | 任务取消完成 |
| `task_failed` | 任务失败 |
| `task_completed` | 任务完成 |
| `task_rerun_created` | 基于旧任务创建重新运行任务 |

建议索引：

| 索引字段 | 中文说明 |
|---|---|
| `task_id, created_at` | 查询任务时间线 |
| `event_type, created_at` | 排查某类事件 |

## 8. 错误与日志存储

错误信息分层保存：

- `tasks.error_summary`：普通用户可见的脱敏摘要。
- `tasks.admin_error_detail`：管理员可见的关键错误详情和必要堆栈摘要。
- `task_files.task_log`：完整任务日志文件索引。
- `task_events.detail_json`：结构化事件详情。

完整日志不直接塞进数据库，避免数据库膨胀。

## 9. 权限判断

普通用户访问任务时：

```text
tasks.created_by = 当前用户 ID
```

管理员访问任务时：

```text
users.role = admin
```

前端按钮状态只用于展示，所有关键操作都必须在后端重新校验：

- 查看任务详情。
- 查看输入文件。
- 下载 Excel。
- 取消任务。
- 重新运行任务。
- 查看管理员错误详情。
- 查看管理员文件。

## 10. SQLite 到 PostgreSQL 的迁移口径

MVP 使用 SQLite，迁移 PostgreSQL 时：

- `TEXT` JSON 字段可迁移为 `JSONB`。
- ISO 时间字符串可迁移为 `TIMESTAMPTZ`。
- 字符串 ID 可继续保留。
- 文件路径仍保存相对路径。
- `data/tasks/` 文件目录仍需独立迁移和备份。

不建议因为未来可能迁移 PostgreSQL，在 MVP 阶段提前引入复杂数据库特性。

## 11. 后续扩展

新增模块时复用通用表：

```text
tasks
task_files
task_events
```

模块专属字段放在独立扩展表，例如：

```text
ops_task_details
```

模块扩展表通过 `task_id` 关联 `tasks.id`，不修改通用任务表结构。

后续如需要页面维护系统资料，可新增：

```text
system_profiles
system_profile_files
```

首版系统资料优先使用文件目录管理，不进入数据库。
