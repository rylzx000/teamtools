# 服务器部署

## 部署原则

- 单机部署。
- 推荐使用 Docker Compose。
- 首版不使用 Nginx，直接通过应用端口访问。
- 首版数据库使用 SQLite，数据库文件挂载到宿主机目录。
- 前端静态文件由 FastAPI 提供。
- 运行数据放在工程根目录下的 `data/`。
- 日志放在工程根目录下的 `logs/`。
- 不在服务器全局 Python 环境中安装项目依赖。

## 推荐部署形态

首版推荐一个应用容器：


访问方式：

```text

```

首版不需要独立后台任务容器，不需要 Redis，不需要 PostgreSQL。

## Docker Compose 文件

建议在工程根目录放置：

```text
/data/teamtools/docker-compose.yml
```

建议服务：

```yaml
services:
  teamtools:
    build: .
    container_name: teamtools
    restart: unless-stopped
    ports:
      - "8000:8000"
    env_file:
      - .env
    volumes:
      - ./data:/app/data
      - ./logs:/app/logs
```

说明：

- `./data:/app/data` 用于保存上传文件、任务结果、SQLite 数据库和运行产物。
- `./logs:/app/logs` 用于保存应用日志。
- 容器重建不会删除 `data/` 和 `logs/`。
- `.env` 只保存非敏感或部署必要配置；不要把用户模型 API Key 写入公共前端代码。

## 模型直连前置检查

首版由用户浏览器调用 DeepSeek 等外部模型。部署前必须在目标内网访问环境验证：

```text
浏览器页面 -> 模型 API
```

检查项：

- CORS 不拦截浏览器请求。
- API Key 能按模型入口要求传入。
- 浏览器能拿到完整响应。
- 网络失败、鉴权失败、限流失败时，前端能拿到可脱敏展示的错误信息。

当前已通过 DeepSeek 浏览器直连最小测试，确认 API 调用流程可连通。正式部署前使用有效账号再验证一次成功响应即可。

如果浏览器不能直连模型 API，不要继续按当前前端直连方案开发，应先确认企业代理入口或改为后端模型代理方案。

## Dockerfile 原则

Dockerfile 后续随代码实现补齐，原则如下：

- 使用 Python 运行后端。
- 如前端在镜像内构建，需要安装 Node.js 并执行前端构建。
- 如前端提前构建好，也可以只复制 `frontend/dist`。
- 安装 Excel 生成所需 Python 依赖。
- 容器启动 FastAPI。

容器启动命令示例：

```text
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

实际模块路径以后端代码为准。

## 首次部署步骤

1. 确认服务器已安装 Docker 和 Docker Compose。
2. 将项目源码放到 `/data/teamtools`。
3. 在 `/data/teamtools` 下创建 `data/` 和 `logs/`。
4. 配置 `.env`。
5. 构建并启动容器。
6. 访问健康检查接口。
7. 使用浏览器访问平台页面。

示例命令：

```bash
cd /data/teamtools
mkdir -p data logs
docker compose build
docker compose up -d
docker compose ps
```

健康检查：

```text
http://服务器IP:8000/api/health
```

本机服务器内检查：

```bash
curl http://127.0.0.1:8000/api/health
```

## 更新部署

代码更新后：

```bash
cd /data/teamtools
docker compose build
docker compose up -d
docker compose ps
```

查看日志：

```bash
docker compose logs -f teamtools
```

停止服务：

```bash
docker compose down
```

注意：`docker compose down` 不会删除挂载在宿主机的 `data/` 和 `logs/`，但不要手动删除这些目录。

## 运行依赖

服务器需要：

- Docker
- Docker Compose

如果前端不在服务器构建，而是在本地构建后上传，则服务器不需要单独安装 Node.js。

如果使用 Docker 部署，服务器不需要单独安装项目 Python 依赖。

## 数据库选择

MVP 使用 SQLite：

```text
/data/teamtools/data/teamtools.db
```

适用原因：

- 首批用户约 10 人。
- 并发不高。
- 任务主要是上传、回传 AI 结果、生成 Excel 和下载。
- 部署和备份简单。

后续如推广到部门、并发增加或需要更强审计能力，可切换 PostgreSQL。

PostgreSQL 后续可以加入 `docker-compose.yml`，不需要手工安装到服务器系统中。

## 文件持久化

以下目录必须保留在宿主机：

```text
/data/teamtools/data
/data/teamtools/logs
```

`data/` 建议包含：

```text
data/
├─ teamtools.db
├─ modules/
└─ tasks/
   └─ fpa/
      └─ {task_id}/
         ├─ input/
         ├─ ai/
         ├─ output/
         └─ runtime/
```

`tasks/` 保存任务输入、AI 请求包、AI 调用结果、生成过程和 Excel 输出。

不要把上传文件、AI 产物、Excel 输出、SQLite 数据库只保存在容器内部。

## 对外访问

首版直接端口访问：

```text
http://服务器IP:8000
```

暂不配置 Nginx。

如后续需要统一域名、HTTPS、访问日志、反向代理或多服务路由，再引入 Nginx 或网关。

## 安全与权限

建议：

- `root` 负责安装 Docker、创建目录和部署。
- 应用在容器内运行。
- 宿主机只暴露必要端口。
- `data/` 和 `logs/` 目录只给部署相关用户和 Docker 进程访问。
- 不把模型 API Key 写死到前端代码。
- 用户自己的模型 API Key 首版保存在浏览器本地。
- `.env` 不提交到代码仓库。

## 备用：无 Docker 直接部署

如果 Docker 不可用，可以使用直接部署作为备用方案。

直接部署需要服务器安装：

- Python 3.11+
- Node.js 20+，仅服务器构建前端时需要
- systemd

直接部署运行方式：

```text
FastAPI 由 systemd 管理
前端静态文件由 FastAPI 托管
SQLite 文件放在 /data/teamtools/data/teamtools.db
```

服务示例：

```text
teamtools-api.service
```

当前推荐仍是 Docker Compose；直接部署只作为 Docker 不可用时的备选方案。
