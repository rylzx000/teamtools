# FPA 数据与文件设计

## 1. 设计原则

- 数据库保存任务状态、权限判断、结果索引和必要摘要。
- 大段输入、过程文件、日志和结果文件保存在运行数据目录。
- 代码目录和运行数据目录分离。
- 用户输入先保存为文件，再由后端读取受控文件生成 AI 请求包。
- 成功任务保留结果文件和必要过程文件。
- 失败任务保留排查文件。
- 已取消任务清理临时文件，只保留任务记录和取消日志摘要。

FPA 表结构引用平台级数据库设计，详见 [05-数据库设计.md](../../architecture/05-数据库设计.md)。FPA 模块只使用 `fpa_task_details` 保存专属字段，通用任务、文件和事件数据复用平台统一表。

## 2. 运行数据目录

本地开发默认运行数据目录：

```text
D:\project\teamtools\data
```

服务器默认运行数据目录：

```text
/data/teamtools/data
```

FPA 建议目录结构：

```text
data/
  config/
    modules/
      fpa/
        module.yaml
        systems.yaml
  modules/
    fpa/
      examples/
      knowledge/
        claimcar/
          source/
          teamtools-system-brief.md
          teamtools-system-index.json
        claimoth/
          source/
          teamtools-system-brief.md
          teamtools-system-index.json
        onlineclaim/
          source/
          teamtools-system-brief.md
          teamtools-system-index.json
        clqp/
          source/
          teamtools-system-brief.md
          teamtools-system-index.json
  tasks/
    fpa/
      {task_id}/
        input/
          pasted.md
          uploaded.md
          merged_input.md
        ai/
          AI请求包.json
          AI请求摘要.json
          AI原始响应.json
          AI调用错误.json
          AI结构化结果.json
          AI分析.md
        runtime/
          Excel脚本输入payload.json
          FPA生成过程.json
          task.log
        output/
          需求名称-FPA工作量评估.xlsx
```

说明：

- `input/` 保存用户原始输入和合并后的输入。
- `ai/` 保存 AI 请求包、模型原始响应、结构化结果、AI 分析说明和调用错误。
- `runtime/` 保存后端生成的 Excel 脚本输入 payload、脚本生成过程、任务日志、临时文件和排查文件。
- `output/` 保存用户可下载的 Excel。
- 成功任务保留 AI 请求摘要、`AI结构化结果.json`、`AI分析.md`、`FPA生成过程.json` 和 Excel。
- 失败任务保留 `ai/` 和 `runtime/` 下的排查文件。
- 已取消任务可清理 `runtime/` 和 `output/` 临时文件。

## 3. 系统资料配置

系统资料首版使用服务器文件配置，不做页面维护。

建议配置文件：

```text
data/config/modules/fpa/systems.yaml
```

配置示例：

```yaml
systems:
  - code: claimcar
    name: 车险理赔核心系统
    system_type: 核心系统
    enabled: true
    sort_order: 10
    knowledge_dir: modules/fpa/knowledge/claimcar
    brief_file: teamtools-system-brief.md

  - code: claimoth
    name: 非车险理赔核心系统
    system_type: 核心系统
    enabled: true
    sort_order: 20
    knowledge_dir: modules/fpa/knowledge/claimoth
    brief_file: teamtools-system-brief.md

  - code: onlineclaim
    name: 在线理赔服务平台
    system_type: 多流程支持系统
    enabled: true
    sort_order: 30
    knowledge_dir: modules/fpa/knowledge/onlineclaim
    brief_file: teamtools-system-brief.md

  - code: clqp
    name: 零配件报价系统
    system_type: 专业工具
    enabled: true
    sort_order: 40
    knowledge_dir: modules/fpa/knowledge/clqp
    brief_file: teamtools-system-brief.md
```

字段说明：

| 字段 | 说明 |
|---|---|
| `code` | 系统唯一编码，用于任务记录和资料定位 |
| `name` | 页面展示名称 |
| `system_type` | 系统类型，用于辅助理解和后续筛选 |
| `enabled` | 是否在页面可选 |
| `sort_order` | 页面排序 |
| `knowledge_dir` | 系统资料目录，相对 `TEAMTOOLS_DATA_DIR`，允许为空 |
| `brief_file` | 精简系统知识包文件名，默认 `teamtools-system-brief.md` |

首批系统编码固定如下，代码、配置、目录名和文档示例必须保持一致，不得自行改名或另造别名：

| 系统编码 | 系统名称 | 资料目录 |
|---|---|---|
| `claimcar` | 车险理赔核心系统 | `modules/fpa/knowledge/claimcar` |
| `claimoth` | 非车险理赔核心系统 | `modules/fpa/knowledge/claimoth` |
| `onlineclaim` | 在线理赔服务平台 | `modules/fpa/knowledge/onlineclaim` |
| `clqp` | 零配件报价系统 | `modules/fpa/knowledge/clqp` |

如后续需要“其他系统”无资料模式，必须先确认对应系统编码；不要在代码或配置中临时使用 `other`、`other_system`、`claim_core`、`online_claim` 等未确认命名。

系统资料目录约定：

| 路径 | 说明 | 后端默认读取 |
|---|---|---:|
| `modules/fpa/knowledge/{system_code}/teamtools-system-brief.md` | 精简系统知识包，默认进入 AI 请求包 | 是 |
| `modules/fpa/knowledge/{system_code}/teamtools-system-index.json` | 可选结构化索引，后续检索或路由使用 | 否 |
| `modules/fpa/knowledge/{system_code}/source/` | 完整系统资料包或原始蒸馏来源 | 否 |

后端读取精简知识包时，路径固定为：

```text
TEAMTOOLS_DATA_DIR + systems.yaml.knowledge_dir + systems.yaml.brief_file
```

读取和异常规则：

- `knowledge_dir` 为空时，进入无资料模式，`no_knowledge_mode = 1`。
- `knowledge_dir` 非空但 `brief_file` 缺失时，任务失败，记录为系统资料配置错误，不静默降级为无资料模式。
- `brief_file` 超过 `09-系统资料蒸馏规则.md` 定义的长度上限时，任务失败或进入管理员可见配置错误，不自动截断。
- 生成 `AI请求包.json` 时默认只读取 `teamtools-system-brief.md`，不读取 `source/` 下的完整资料包。

系统类型首版不强制枚举，推荐值：

- 核心系统
- 多流程支持系统
- 专业工具
- 其他
- 未知系统

## 4. 模块配置

FPA 模块配置项在实现阶段继续细化，首版至少需要覆盖：

- 模板文件路径。
- 输入字符上限。
- 上传文件大小上限。
- 示例输入和示例输出目录。

建议配置文件：

```text
data/config/modules/fpa/module.yaml
```

MVP 不维护模板版本和系统资料版本。后续如果需要追溯，再增加版本字段。

## 5. 输入限制与输入文件

建议把输入限制作为 FPA 模块配置集中维护。

建议配置项：

| 配置项 | 建议值 | 说明 |
|---|---:|---|
| `max_input_chars` | `20000` | 粘贴内容和上传 Markdown 的有效内容字符上限 |
| `max_upload_bytes` | `262144` | 上传 `.md` 文件大小上限，等于 256KB |
| `target_person_days_scale` | `1` | 目标人天最多 1 位小数 |
| `default_count_timing` | `估算早期` | 页面未选择规模计数时机时使用的默认值 |

任务输入目录固定保存以下文件角色：

| 文件 | 生成条件 | 说明 |
|---|---|---|
| `input/pasted.md` | 用户提供粘贴文本 | 原始粘贴内容 |
| `input/uploaded.md` | 用户上传 `.md` | 原始上传文件内容 |
| `input/merged_input.md` | 每个任务都生成 | 后端生成 AI 请求包时读取的统一输入文件 |

当粘贴文本和上传文件同时存在时，`merged_input.md` 按“粘贴文本在前、上传文件在后”合并。

## 6. 任务主数据

FPA 任务可以复用平台统一任务表，并通过 `module = fpa` 区分模块。

建议任务字段：

| 字段 | 说明 |
|---|---|
| `id` | 任务 ID |
| `module` | 模块编码，FPA 固定为 `fpa` |
| `created_by` | 提交人 |
| `title` | 任务名称 |
| `system_code` | 用户选择的系统编码 |
| `system_name` | 提交时的系统名称快照 |
| `status` | 当前任务状态 |
| `cancel_requested` | 是否请求取消 |
| `target_person_days` | 目标人天，可为空 |
| `count_timing` | 规模计数时机，默认 `估算早期` |
| `result_median_person_days` | 结果中值，可为空 |
| `target_hit` | 是否命中目标，可为空 |
| `quality_flags` | 轻量质量提示标记 |
| `no_knowledge_mode` | 是否无资料模式 |
| `retry_count` | 前端模型调用重试次数或重新提交次数，可为空 |
| `failure_stage` | 失败阶段 |
| `error_summary` | 脱敏错误摘要 |
| `admin_error_detail` | 管理员错误详情 |
| `rerun_from_task_id` | 重新运行来源任务 ID，可为空 |
| `created_at` | 提交时间 |
| `started_at` | 开始时间 |
| `finished_at` | 完成时间 |
| `cancelled_at` | 取消时间 |
| `cancelled_by` | 取消人 |

`result_median_person_days` 和 `target_hit` 来自后台脚本计算结果，不依赖读取 Excel 公式缓存值。

失败任务重新运行时，新任务通过 `rerun_from_task_id` 关联原失败任务，原失败任务记录不覆盖。

## 7. 任务文件索引

建议用任务文件索引记录不同产物路径，避免把路径硬编码在业务逻辑中。

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 文件记录 ID |
| `task_id` | 任务 ID |
| `file_role` | 文件角色 |
| `path` | 文件路径 |
| `downloadable` | 是否允许下载 |
| `created_at` | 创建时间 |

FPA 文件角色建议：

| file_role | 说明 | 普通用户权限 |
|---|---|---|
| `input_pasted` | 粘贴文本保存文件 | 仅复制重新生成时使用 |
| `input_uploaded` | 上传 Markdown 文件 | 仅复制重新生成时使用 |
| `input_merged` | 合并后的输入文件 | 仅复制重新生成时使用 |
| `ai_request_package` | `AI请求包.json` | 普通用户通过 `ai-request` 专用接口获取，不在文件列表展示、不提供下载；管理员可查看排查文件 |
| `ai_request_summary` | `AI请求摘要.json` | 默认管理员查看；普通用户不展示 |
| `ai_raw_response` | `AI原始响应.json` | 管理员查看 |
| `ai_call_error` | `AI调用错误.json` | 普通用户只看脱敏错误摘要；管理员可查看完整错误文件 |
| `ai_structured_json` | `AI结构化结果.json` | 管理员查看 |
| `ai_analysis_md` | `AI分析.md` | 页面查看、复制 |
| `fpa_process_json` | `FPA生成过程.json` | 页面查看、复制 |
| `excel_script_payload` | `Excel脚本输入payload.json` | 管理员查看 |
| `excel_result` | Excel 结果文件 | 下载 |
| `task_log` | 任务日志 | 管理员查看 |

## 8. JSON 产物结构原则

FPA 链路中存在三类 JSON：`AI结构化结果.json`、Excel 脚本输入 payload、`FPA生成过程.json`。

结构原则：

- AI JSON 字段以功能点明细业务判断为主。
- AI JSON 不输出汇总、人天计算、目标命中判断、Excel 单元格位置、模板路径、输出路径或脚本字段 `target_work_days`。
- 平台/接口/任务表统一使用 `target_person_days`。
- `target_person_days` 可以进入 AI 请求包正文作为参考目标，但不属于 AI JSON 输出字段。
- 后端生成 Excel 脚本输入 payload 时，把 `target_person_days` 映射为现有脚本兼容字段 `target_work_days`。
- 后端生成 Excel 脚本输入 payload 时，合并 AI JSON、任务配置、模板路径、输出路径、目标人天和兜底后的 `requirement_name`。
- 脚本从 Excel 脚本输入 payload 生成 `FPA生成过程.json`。
- `FPA生成过程.json` 可包含标准化明细、脚本计算结果、目标命中、质量提示、模板路径、输出路径和过程信息。
- 解释文本用于页面查看和人工理解。
- 详细契约见 [07-AI输出与提示词契约设计.md](07-AI输出与提示词契约设计.md)。

## 9. 任务事件

任务事件用于记录状态变化和关键操作。

建议事件：

- 任务创建。
- AI 请求包生成。
- 前端获取 AI 请求包。
- 前端回传 AI 结果。
- AI 调用失败。
- 状态变更。
- 进入无资料模式。
- JSON 校验失败。
- Excel 生成完成。
- 用户请求取消。
- 后端完成取消。
- 失败任务重新运行。
- 任务失败。
- 任务完成。

事件内容应支持普通用户脱敏展示和管理员排查。

## 10. 输入合并规则

当用户同时提供粘贴内容和上传 `.md` 文件时，后台生成 `merged_input.md`。

建议合并格式：

```markdown
### 粘贴输入

...

### 上传 Markdown 输入

...
```

重新生成时，优先把 `merged_input.md` 带回输入框。

## 11. 结果命名

Excel 命名规则：

```text
{需求名称}-FPA工作量评估.xlsx
```

当需求名称为空时：

1. 使用系统总结的短需求名称。
2. 使用上传文件名、需求文本摘要或任务编号生成兜底名称。
3. 仍无法总结时，使用 `FPA工作量评估-YYYYMMDD-HHmm.xlsx`。

同名任务不覆盖旧文件。实际保存时可以在文件名或目录层面追加 `task_id` 保证唯一。

## 12. FPA 资源目录

FPA 模板、skill、JSON schema、Excel 映射和轻量规则配置放在数据目录：

```text
data/modules/fpa/profile/
  profile.yaml
  skill/
  templates/
  schema/
  mapping/
  rules/
```

说明：

- `profile.yaml` 只记录当前使用的资源路径和开关，MVP 不记录版本号。
- Excel 生成依赖 `templates/` 下的当前模板。
- 一个 Excel 模板对应一个 `mapping/excel_mapping.yaml`。
- 模板结构变化时，需要同步维护映射文件。
- skill、模板、schema、mapping 和 rules 由管理员手动替换。
- 替换资源文件前建议暂停新任务提交或在低峰期处理，避免请求包生成时读取到不一致资源。
- MVP 不维护历史版本、归档目录或任务资源快照。
- 脚本实现属于平台代码，后续实现阶段接入，但脚本只能按资源配置和映射生成确定性结果。

详细约定见 [06-资源包与生成契约设计.md](06-资源包与生成契约设计.md)。

现有 FPA 资产来源目录：

```text
D:\project\fpa功能点法工作量评估
```

后续实现阶段从该目录确认模板、脚本、规约和示例文件。

## 13. 验收样例目录

首版需要保留开发与验收样例，但不放到前端页面。

建议目录：

```text
data/modules/fpa/examples/
  input/
  expected/
```

样例用于验证流程和产物，不用于人工抽查评估准确性。

## 14. 计算规则维护

模板公式不是只存在于 Excel 文件里的隐式规则。

维护要求：

- 关键公式需要在 [04-FPA计算规则.md](04-FPA计算规则.md) 中写明。
- 脚本中的计算逻辑需要集中实现。
- 模板公式调整时，同步更新规则文档和脚本。
- 规则文档、脚本和模板三者不一致时，以人工确认后的规则文档作为修正基线。

## 15. AI 请求包读取边界

后端生成 AI 请求包时，只允许读取以下目录：

- 当前任务目录。
- 当前任务选择的系统资料目录。
- FPA 资源目录。

不允许读取其他任务目录、平台敏感配置、模型 key、环境变量或系统敏感路径。请求包中不得暴露服务器绝对路径和用户无权查看的内部配置。

## 16. 查看与下载权限

普通用户：

- 可查看自己任务的状态、摘要、错误详情和结果。
- 可下载自己任务的 Excel。
- 可页面查看和复制自己的 `AI分析.md`。
- 可页面查看和复制自己的 `FPA生成过程.json`。
- 不可下载 Markdown 和 JSON 过程文件。
- `quality_gate.failed` 或 `deliverable_valid=false` 表示强风险或不建议直接交付，默认不阻断 Excel 下载；只有结构错误、模板错误、脚本错误、JSON 校验失败等技术失败才阻断成功产物生成。

管理员：

- 可查看全部任务。
- 可查看完整任务日志和排查文件。
- 可查看失败任务的完整错误详情。

所有用户：

- 不提供历史任务删除入口。
- 不暴露模型 key、环境变量和服务器敏感路径。
