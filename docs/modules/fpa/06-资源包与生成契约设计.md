# FPA 资源包与生成契约设计

## 1. 设计目标

本文档记录 FPA 模块中模板、skill、JSON schema、Excel 映射和计算规则的接入方式。

MVP 目标是保持实现轻量，同时给后续调整模板和 skill 留出空间。

核心原则：

- 平台代码固定任务流程，不硬编码具体模板细节。
- FPA 资源通过配置文件指定。
- 能由代码或脚本确定生成的内容，不交给 AI 生成。
- AI 只负责需求理解、功能点拆分、业务分类、复杂度判断和解释说明。
- Excel 文件由脚本基于模板生成，不由大模型直接生成。

## 2. 资源目录

FPA MVP 使用一套当前生效资源包，目录固定为：

```text
data/modules/fpa/profile/
  profile.yaml
  skill/
    prompt_template.md
  templates/
  schema/
  mapping/
  rules/
```

目录说明：

| 目录或文件 | 用途 |
|---|---|
| `profile.yaml` | 当前资源包配置，记录启用状态和各资源文件路径 |
| `skill/` | 后端生成 AI 请求包时读取的规则、提示词或说明材料 |
| `skill/prompt_template.md` | 后端生成 AI 请求包时使用的固定提示词模板 |
| `templates/` | Excel 模板文件 |
| `schema/` | AI 输出 JSON 的结构约束 |
| `mapping/` | JSON 字段写入 Excel 的位置和区域映射 |
| `rules/` | 脚本计算所需的参数、映射和轻量规则配置 |

MVP 不维护以下内容：

- `current/` 目录。
- `archive/` 目录。
- profile 历史版本。
- skill 历史版本。
- template 历史版本。
- 每个任务的资源包快照。

后续如果需要审计、复现或版本追溯，再增加归档目录和版本字段。

## 3. profile.yaml

`profile.yaml` 只记录当前使用的资源路径和开关，不记录版本号。

示例：

```yaml
enabled: true

skill:
  path: skill/
  prompt_template: skill/prompt_template.md

template:
  path: templates/fpa工作量评估模版.xlsx

schema:
  path: schema/result.schema.json

mapping:
  path: mapping/excel_mapping.yaml

calculation:
  path: rules/calculation.yaml
```

处理规则：

- 路径相对于 `data/modules/fpa/profile/`。
- 后端每次生成 AI 请求包或 Excel 结果时读取当前配置。
- 配置文件缺失或关键路径缺失时，任务失败，并记录明确错误。
- `skill/prompt_template.md` 缺失时任务失败，不使用隐藏默认提示词。
- MVP 不做 profile 热加载管理页。
- 管理员通过服务器文件替换和修改配置。

## 4. 模板与映射关系

MVP 采用一个 Excel 模板对应一个映射文件。

```text
templates/fpa工作量评估模版.xlsx
mapping/excel_mapping.yaml
```

规则：

- 平台不自动识别 Excel 结构变化。
- 模板结构变化时，管理员或开发人员必须同步维护 `excel_mapping.yaml`。
- Excel 生成脚本只根据映射文件写入约定单元格和明细区域。
- 脚本不主动覆盖公式单元格。
- 脚本需要保留模板工作表、格式、公式和参数区。
- 功能点条目超过模板默认行数时，由脚本动态插入行，不拆分多个 Excel。

Excel 映射和脚本生成细节见 [08-Excel映射与脚本生成设计.md](08-Excel映射与脚本生成设计.md)。

## 5. AI 与脚本职责边界

FPA 生成链路按确定性拆分职责。

AI 负责：

- 阅读用户需求。
- 结合系统资料理解业务背景。
- 拆分功能点明细。
- 判断每条功能点的类型、复杂度或权重相关依据。
- 输出每条功能点的判断理由。
- 参考目标人天，在 FPA 框架内对拆分粒度、复用程度和修改类型做可解释校准。
- 输出需求摘要、风险、不确定项和分析说明。
- 按 schema 生成明细级业务判断 JSON。

脚本负责：

- 读取前端回传的 `AI结构化结果.json`。
- 校验 JSON schema。
- 标准化字段。
- 生成功能点序号。
- 计算功能点汇总。
- 计算结果人天、中值或范围。
- 判断是否命中目标人天。
- 生成页面摘要字段。
- 生成质量提示中可确定的部分。
- 根据 Excel 映射填充模板。
- 动态插入明细行。
- 生成最终 Excel 文件名。

AI 不负责：

- 生成 Excel。
- 生成 Excel 单元格位置。
- 计算汇总值。
- 计算结果人天。
- 判断是否命中目标。
- 为贴近目标人天反推、凑值、无依据新增功能点或修改系统属性。
- 生成条目序号。
- 生成最终文件名。
- 输出脚本可以确定生成的质量提示。

## 6. JSON 输出契约

AI 输出的 JSON 是脚本生成 Excel 的唯一结构化输入来源。

规则：

- JSON 必须是合法 JSON。
- JSON 必须通过 `schema/result.schema.json` 校验。
- 核心字段严格限制。
- 未在 schema 中声明的核心字段不参与处理。
- 关键字段缺失、类型错误或枚举错误时，任务失败。

设计要求：

- schema 只要求 AI 输出明细级业务判断字段。
- schema 不要求 AI 输出汇总、计算、命中判断等可由脚本确定的字段。
- schema 可以保留少量自由说明字段，例如 `analysis_notes`、`uncertainties`。
- 自由说明字段只用于页面展示和人工理解，不用于 Excel 确定性计算。

## 7. JSON 校验失败

当 JSON 解析或 schema 校验失败时：

- 任务状态改为失败。
- 失败阶段记录为结果校验。
- 普通用户看到脱敏后的错误原因和建议操作。
- 管理员可查看完整校验错误、AI 原始响应、结构化结果和任务日志。
- 后端不自动调用模型修复；如果需要重试，由前端重新调用模型并重新回传，或用户复制任务重新生成。

## 8. Excel 生成契约

Excel 生成只能由脚本完成。

输入：

- `FPA生成过程.json`
- `profile.yaml`
- `excel_mapping.yaml`
- `calculation.yaml`
- Excel 模板文件

输出：

- `结果.xlsx`
- 页面展示需要的摘要字段
- 质量提示

文件落地规则：

```text
AI分析.md.tmp -> AI分析.md
AI结构化结果.json.tmp -> AI结构化结果.json
FPA生成过程.json.tmp -> FPA生成过程.json
结果.xlsx.tmp -> 结果.xlsx
```

`.tmp` 文件不登记为正式产物。正式文件生成成功后，再写入文件索引和任务结果字段。

## 9. 计算规则

页面展示结果不能依赖读取 Excel 公式缓存值。

处理规则：

- Excel 模板公式保留，供用户打开文件后自动重算和人工调整。
- 后台脚本按模板公式口径同步计算页面关键结果。
- 脚本计算逻辑需要集中实现。
- `calculation.yaml` 只承载参数、字段映射或轻量规则。
- 不把复杂公式全部写成 YAML 规则。
- 模板公式变化时，需要同步更新 [04-FPA计算规则.md](04-FPA计算规则.md)、脚本计算逻辑和必要配置。

## 10. 资源更新方式

MVP 通过手工替换服务器文件更新 FPA 资源。

更新步骤：

1. 暂停新任务提交或选择低峰期操作。
2. 替换 `data/modules/fpa/profile/` 下的模板、skill、schema、mapping 或 rules 文件。
3. 如有路径变化，同步修改 `profile.yaml`。
4. 检查文件路径是否存在。
5. 提交测试任务验证流程。

约束：

- 不在请求包生成或 Excel 生成过程中直接覆盖资源文件。
- API 可以不停，但建议避免同一时刻创建新任务。
- 已生成请求包的任务不随资源替换自动变化。
- 已完成任务不受资源替换影响，因为结果文件已经保存到任务目录。

## 11. 与系统资料的关系

FPA 资源包只描述 FPA 评估方法、模板、输出结构和脚本映射。

系统资料仍按系统配置独立维护：

```text
data/modules/fpa/knowledge/
  claimcar/
    source/
    teamtools-system-brief.md
    teamtools-system-index.json
  claimoth/
    source/
    teamtools-system-brief.md
    teamtools-system-index.json
  onlineclaim/
    source/
    teamtools-system-brief.md
    teamtools-system-index.json
  clqp/
    source/
    teamtools-system-brief.md
    teamtools-system-index.json
```

首批系统编码和目录名固定如下：

| 系统编码 | 系统名称 | 精简知识包路径 |
|---|---|---|
| `claimcar` | 车险理赔核心系统 | `data/modules/fpa/knowledge/claimcar/teamtools-system-brief.md` |
| `claimoth` | 非车险理赔核心系统 | `data/modules/fpa/knowledge/claimoth/teamtools-system-brief.md` |
| `onlineclaim` | 在线理赔服务平台 | `data/modules/fpa/knowledge/onlineclaim/teamtools-system-brief.md` |
| `clqp` | 零配件报价系统 | `data/modules/fpa/knowledge/clqp/teamtools-system-brief.md` |

代码和配置必须使用以上编码，不得自行引入 `claim_core`、`online_claim`、`other_system` 等别名。

后端读取规则：

- 先根据任务的 `system_code` 读取 `data/config/modules/fpa/systems.yaml`。
- 再拼接 `TEAMTOOLS_DATA_DIR + knowledge_dir + brief_file` 得到精简知识包路径。
- `brief_file` 未配置时按 `teamtools-system-brief.md` 处理。
- `knowledge_dir` 为空时按无资料模式处理。
- `knowledge_dir` 非空但精简知识包不存在时，任务失败并记录系统资料配置错误。
- 精简知识包超过长度上限时，任务失败或记录管理员可见配置错误，不静默截断。
- `source/` 目录只作为完整资料包或蒸馏来源，不默认进入 AI 请求包。

后端生成 AI 请求包时使用：

- 用户输入正文。
- 已选系统的 `teamtools-system-brief.md`。
- FPA 资源包中的提示词模板和输出契约。
- 是否无资料模式。
- 目标人天参考值。

平台不默认把系统资料全文预先拼接进请求包，避免 token 过大。MVP 阶段不做资料片段检索，默认只拼接精简系统知识包。

## 12. 后续扩展

以下能力不进入 MVP，但目录和职责边界允许后续增加：

- profile 版本号。
- 模板版本号。
- skill 版本号。
- 历史归档目录。
- 任务资源快照。
- 多套 FPA profile 切换。
- 资源管理页面。
- 自动升级或回滚脚本。
- schema 兼容性检查工具。
- mapping 与模板结构一致性检查工具。
