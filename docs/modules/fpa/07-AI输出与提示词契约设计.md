# FPA AI 输出与提示词契约设计

## 1. 设计目标

本文档定义 FPA 模块中 AI 输出 JSON、脚本输入、页面可控参数和提示词模板的边界。

核心原则：

- AI 只输出明细级业务判断。
- 汇总、计算、Excel 填充、命中目标判断和文件命名由脚本完成。
- `target_person_days` 可以作为参考目标进入提示词，供 AI 在 FPA 框架内做可解释校准。
- `result.schema.json` 只约束前端回传的 AI 结构化 JSON。
- `FPA生成过程.json` 是脚本生成产物，不由 AI 直接生成完整文件。
- 页面可控参数优先于 AI 判断。
- 提示词模板由资源包提供，后端生成 AI 请求包时读取模板，不在代码中硬编码完整提示词。

## 2. 产物边界

FPA 处理过程中存在三类 JSON，必须明确区分。

| 产物 | 生成方 | 是否受 `result.schema.json` 约束 | 用途 |
|---|---|---:|---|
| `AI结构化结果.json` | AI | 是 | 保存功能点明细、项目特征建议、审计说明等业务判断 |
| Excel 脚本输入 payload | 后端 | 否 | 后端合并 AI 结构化结果、任务配置、模板路径、输出路径、目标人天后调用脚本 |
| `FPA生成过程.json` | 脚本 | 否 | 保存标准化明细、计算结果、质量提示、输出路径和过程信息 |

`result.schema.json` 只校验 AI 结构化 JSON。脚本输入 payload 和 `FPA生成过程.json` 由后端和脚本生成，不要求 AI 输出其中的路径、目标人天、计算字段或 Excel 单元格字段。

## 3. AI JSON 顶层结构

MVP 的 AI JSON 顶层字段如下：

```json
{
  "requirement_name": "需求名称",
  "assessment_context": {},
  "project_features": {},
  "items": [],
  "analysis_notes": "",
  "uncounted_items": [],
  "quality_notes": [],
  "coverage_notes": "",
  "uncertainties": []
}
```

字段说明：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `requirement_name` | 否 | AI 总结的需求名称。前端传入标题优先，缺失时才使用 AI 总结 |
| `assessment_context` | 否 | 评估上下文，例如系统类型、需求等级、估算模式、资料等级 |
| `project_features` | 否 | AI 看到的项目特征参考值，不作为最终参数来源 |
| `items` | 是 | 功能点明细数组 |
| `analysis_notes` | 否 | AI 分析说明，供页面展示和人工理解 |
| `uncounted_items` | 否 | 未计数候选项数组，用于说明纯 UI、字段、按钮、日志等未形成独立功能点的原因 |
| `quality_notes` | 否 | 结构化质量提示数组，用于人工复核拆分过粗、覆盖不足或复用/修改类型偏紧等风险 |
| `coverage_notes` | 否 | 覆盖说明，概述本次需求片段覆盖情况和未覆盖原因 |
| `uncertainties` | 否 | 不确定点列表，供页面展示和人工复核 |

规则：

- `items` 必须是非空数组。
- AI 输出不是 Excel 脚本输入 payload；后端再把 AI 结构化结果、任务参数和模板配置转换为 Excel payload。
- 顶层不要求 AI 输出汇总值。
- 顶层不要求 AI 输出人天结果。
- 顶层不要求 AI 输出目标命中判断。
- 顶层不要求 AI 输出 Excel 路径、文件名或单元格位置。
- 顶层不允许 AI 输出 `template_path`、`output_path`、`target_work_days`、`target_hit`、`adjusted_work_days_middle`、`ufp`、`us`、`adjusted_fp` 等 Excel 计算字段。
- 平台字段 `target_person_days` 由任务配置提供，可进入提示词作为参考目标，但不要求 AI 输出。

## 4. 功能点明细字段

`items` 每条功能点沿用现有 FPA 脚本字段。

```json
{
  "system": "系统",
  "level1_module": "一级模块",
  "level2_module": "二级模块",
  "level3_module": "三级模块",
  "level4_module": "四级模块",
  "function_description": "功能描述",
  "count_item_name": "计数项名称",
  "category": "EI",
  "reuse": "中",
  "change_type": "修改",
  "remark": "判断依据"
}
```

必填字段：

| 字段 | 说明 |
|---|---|
| `system` | 功能点所属系统，必须输出系统中文名，例如 `在线理赔服务平台`，不得输出 `onlineclaim` 等系统编码 |
| `level1_module` | 一级模块 |
| `function_description` | 功能描述 |
| `count_item_name` | 计数项名称 |
| `category` | 功能点类别 |
| `reuse` | 重用程度 |
| `change_type` | 修改类型 |
| `remark` | 判断依据 |

可为空字段：

- `level2_module`
- `level3_module`
- `level4_module`

## 5. 枚举值

MVP 沿用现有脚本枚举。

| 字段 | 允许值 |
|---|---|
| `category` | `ILF`、`EIF`、`EI`、`EO`、`EQ` |
| `reuse` | `高`、`中`、`低` |
| `change_type` | `新增`、`修改`、`删除` |
| `system` | `车险理赔核心系统`、`非车险理赔核心系统`、`在线理赔服务平台`、`零配件报价系统` |

## 5.1 未计数与复核字段

为了避免 AI 只输出粗略明细，本契约允许 AI 额外输出结构化复核字段，但这些字段不直接写入 Excel 明细。

`uncounted_items` 每项结构：

```json
{
  "description": "未计数候选项",
  "reason": "不计数原因",
  "related_requirement_section": "关联需求片段"
}
```

`quality_notes` 每项结构：

```json
{
  "code": "REVIEW_STAGE_PREVIEW_SPLIT",
  "message": "需要人工复核的质量提示",
  "severity": "medium"
}
```

`severity` 只能是 `low`、`medium`、`high`。`coverage_notes` 是字符串，用于说明提交、保存、阶段预览、缺项提示、配置查询、补拍/跳过、继续提交等需求片段是否已覆盖或为何未计数。

处理规则：

- `category` 由 AI 判断，脚本校验。
- `reuse` 由 AI 判断，脚本校验。
- `change_type` 由 AI 判断，脚本校验。
- 每条功能点必须在 `remark` 中说明类型、复用和修改类型的判断依据。

## 6. assessment_context

`assessment_context` 可选，内部字段也可选。

建议字段：

```json
{
  "system_type": "核心系统",
  "requirement_level": "D2",
  "estimation_mode": "核心复杂改造模式",
  "source_level": "L2"
}
```

规则：

- 有依据时填写。
- 无资料模式或输入信息不足时，不强制 AI 编造。
- 缺少该字段时，脚本仍应能生成 Excel。
- 缺少该字段时，脚本质量提示可以减少或降级。

## 7. 页面可控项目特征

MVP 页面只开放一个项目特征：

| 字段 | 页面标签 | 默认值 | 允许值 |
|---|---|---|---|
| `count_timing` | 规模计数时机 | `估算早期` | 以模板基础参数允许值为准，首批使用 `估算早期`、`估算中期`、`估算晚期` |

处理规则：

- 其他项目特征使用模板默认值。
- 后续如有需要，再将更多项目特征放入高级参数。
- 页面选择值由后端写入 AI 请求包，并在生成 Excel 脚本输入 payload 时与模板默认值合并。
- 脚本最终使用页面选择值和模板默认值合并后的 `project_features`。

## 8. project_features 优先级

`project_features` 的最终来源优先级如下：

1. 用户在页面选择的值。
2. 平台默认值：`规模计数时机 = 估算早期`。
3. 模板默认值。
4. AI JSON 中的 `project_features` 只作为分析参考，不作为最终来源。

如果 AI JSON 中的 `project_features` 与页面选择冲突，以页面选择为准。

平台生成脚本输入 payload 时，应把最终合并后的项目特征写入 payload，供脚本稳定生成 Excel。平台/接口/任务表统一使用 `target_person_days`；调用现有脚本前由后端映射为 payload 中的 `target_work_days`。

## 9. 提示词模板

固定提示词模板放在 FPA 资源包中：

```text
data/modules/fpa/profile/skill/prompt_template.md
```

后端不在代码中硬编码完整提示词，而是读取模板并替换固定变量，生成 AI 请求包。

建议变量：

```text
{{ input_text }}
{{ knowledge_summary }}
{{ no_knowledge_mode }}
{{ target_person_days }}
{{ project_features }}
{{ schema_text }}
```

目标人天使用口径：

- `target_person_days` 是可选参考目标，不是硬性反推值。
- AI 应先基于需求事实、系统资料和 FPA 规则形成基准功能点拆分。
- 目标与需求规模匹配时，AI 可以在功能点拆分粒度、复用程度、修改类型上做可解释校准。
- AI 不输出最终人天、目标命中判断、`target_work_days` 或任何 Excel 计算字段。
- AI 必须要求 `items[].system` 输出系统中文名，不输出系统编码。
- AI 必须逐项识别业务基本过程，避免把阶段预览、最终预览、缺项提示、提交、配置查询、状态更新、允许缺项提交、补拍/继续提交等独立业务动作过度合并。
- 如果候选项不计数，必须写入 `uncounted_items` 或 `uncertainties` 并说明原因。
- 不得出现“凑值”“反推”“为了贴近目标”等表达。
- 不得为了目标人天无依据新增功能点、拆碎字段按钮、降低复用或改变修改类型。

规则：

- 普通用户不能编辑提示词。
- 管理员通过替换服务器文件调整提示词。
- 提示词模板缺失时，任务失败。
- 不使用隐藏默认提示词。
- 提示词模板必须明确要求 AI 输出符合 `result.schema.json` 的结构化 JSON。
- 如果同时输出 Markdown 分析说明，该说明只用于页面查看和人工理解。

## 10. JSON 校验

后端收到前端回传的 `AI结构化结果.json` 后，执行以下检查：

1. 是否为合法 JSON。
2. 是否通过 `result.schema.json` 校验。
3. 是否包含非空 `items`。
4. 每条明细的必填字段和枚举值是否有效。
5. `items[].system` 是否为已知系统中文名。
6. 是否存在 `target_work_days`、`adjusted_work_days_middle`、`ufp`、`us` 等禁止字段。
7. `uncounted_items`、`quality_notes` 和 `coverage_notes` 是否符合结构约束。

失败时：

- 任务失败。
- 保存 AI 原始响应和校验错误。
- 普通用户看到脱敏后的错误原因和建议操作。
- 管理员可以查看完整校验错误和任务日志。

## 11. 与现有 FPA 工具的差异

现有 `fpa-skill` 和脚本可以作为实现基础，但平台 MVP 有以下调整：

- AI 不直接生成完整 `FPA生成过程.json`。
- `FPA生成过程.json` 由平台脚本生成。
- 页面控制 `规模计数时机`，默认 `估算早期`。
- AI JSON 中的 `project_features` 不作为最终参数来源。
- 平台设计不设置 29 条功能点硬上限；目标是同一个 Excel 内通过脚本动态增行支持超过模板默认明细行数。
- Excel 只能由脚本基于模板生成。
