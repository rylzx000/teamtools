# FPA AI 输出与提示词契约设计

## 1. 设计目标

本文档定义 FPA 模块中 AI 输出、脚本输入、页面可控参数和提示词模板的边界。`AI`、`JSON`、`Excel`、`payload` 等英文保留，是既有能力、数据格式、办公软件名称和机器可读数据结构专有名词，避免破坏接口约定。

核心原则：

- AI 只输出业务判断：变更事实、场景路由、拆分/合并决策、冻结功能点清单和复核提示。
- `frozen_items` 是 `AI评估.md`、Excel payload 和 `FPA生成过程.json` 的唯一明细依据。
- 汇总、计算、Excel 填充、目标命中判断和文件命名由后端与脚本完成。
- `target_person_days` 只作为参考目标进入提示词，不得用于反推、凑值或改变冻结清单边界。
- `result.schema.json` 只约束前端回传的 AI 结构化 JSON。
- `FPA生成过程.json` 是脚本生成产物，不由 AI 直接生成。
- 普通用户只看结果摘要和 Excel 下载；`AI评估.md`、结构化 JSON、Excel payload 和过程 JSON 仅供管理员或后台排查。

## 2. 产物边界

FPA 处理过程中存在三类 JSON，必须明确区分。

| 产物 | 生成方 | 是否受 `result.schema.json` 约束 | 用途 |
|---|---|---:|---|
| `AI结构化结果.json` | AI | 是 | 保存评估上下文、事实、路由、拆分/合并、冻结清单和复核提示 |
| Excel 脚本输入 payload | 后端 | 否 | 后端合并 AI 结构化结果、任务配置、模板路径、输出路径和目标人天后调用脚本 |
| `FPA生成过程.json` | 脚本 | 否 | 保存标准化明细、计算结果、质量提示、输出摘要和过程信息 |

边界规则：

- AI 不输出模板路径、输出路径、目标命中、人天结果、Excel 单元格或公式字段。
- 后端把 `frozen_items` 原样映射为脚本 payload 的 `items` 字段；这里的 `items` 只是脚本输入字段名，不是 AI 顶层契约。
- 脚本可以标准化备注和计算结果，但不得新增、删除、拆分、合并、重排或重判功能点。
- 过程 JSON 必须保留 `stable_id`、`fact_ids`、`route_ids`、`system_scene_ids`、`linked_process_ids` 和 `linked_data_ids`，便于管理员复核。

## 3. AI JSON 顶层结构

AI 结构化结果使用 v2 顶层契约：

```json
{
  "schema_version": "fpa.ai_contract.v2",
  "requirement_name": "需求名称",
  "assessment_context": {},
  "project_features": {},
  "change_facts": [],
  "routing_decisions": [],
  "split_merge_decisions": [],
  "frozen_items": [],
  "review_notes": []
}
```

字段说明：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `schema_version` | 是 | 固定为 `fpa.ai_contract.v2` |
| `requirement_name` | 是 | AI 总结的需求名称；前端传入标题仍优先用于任务展示 |
| `assessment_context` | 是 | 当前任务系统、资料模式、评估口径和目标参考等上下文 |
| `project_features` | 否 | AI 看到的项目特征参考值，不作为最终 Excel 参数来源 |
| `change_facts` | 是 | 从需求和系统资料抽取的可追溯变更事实 |
| `routing_decisions` | 是 | 场景路由判断，引用 `change_facts` |
| `split_merge_decisions` | 是 | 拆分、合并、不计数或复核决策，引用路由和冻结项 |
| `frozen_items` | 是 | 最终冻结功能点清单，必须非空 |
| `review_notes` | 否 | 管理员和人工复核提示 |

旧顶层功能点明细契约不再使用；AI 不得另行生成候选清单或与 `frozen_items` 不一致的明细。

## 4. 事实、路由与拆分决策

`change_facts` 用于记录需求中可支持功能点判断的事实。

```json
{
  "fact_id": "F-001",
  "description": "用户提交影像补传资料",
  "source": "需求第 2.1 节",
  "system_scene_ids": ["CAR-CLAIM-001"]
}
```

`routing_decisions` 用于说明事实命中的系统场景和 FPA 类别方向。

```json
{
  "route_id": "R-001",
  "fact_ids": ["F-001"],
  "decision": "按影像补传提交 EI 计数",
  "reason": "外部用户向系统提交并维护业务数据"
}
```

`split_merge_decisions` 用于记录最终拆分、合并、不计数或待复核原因。

```json
{
  "decision_id": "D-001",
  "route_ids": ["R-001"],
  "decision_type": "split",
  "result_stable_ids": ["FP-001"],
  "reason": "提交补传资料形成独立 EI"
}
```

规则：

- `change_facts[].fact_id` 使用 `F-001` 格式。
- `routing_decisions[].route_id` 使用 `R-001` 格式。
- `split_merge_decisions[].decision_id` 使用 `D-001` 格式。
- 所有引用字段必须引用同一 JSON 内已存在编号，不允许悬空引用。
- 不计数候选也必须在路由或拆分/合并决策中说明原因。

## 5. 冻结功能点清单

`frozen_items` 每条功能点使用以下结构：

```json
{
  "stable_id": "FP-001",
  "system": "车险理赔核心系统",
  "level1_module": "理赔资料管理",
  "level2_module": "影像补传",
  "level3_module": "",
  "level4_module": "",
  "function_description": "用户提交影像补传资料",
  "count_item_name": "影像补传提交",
  "category": "EI",
  "reuse": "中",
  "change_type": "新增",
  "remark": "类型依据：外部输入维护补传资料，按 EI；复用依据：复用既有理赔流程，复用中；修改类型依据：新增补传提交能力；关联数据：维护 FP-002 影像补传记录 ILF。",
  "fact_ids": ["F-001"],
  "route_ids": ["R-001"],
  "system_scene_ids": ["CAR-CLAIM-001"],
  "linked_process_ids": [],
  "linked_data_ids": ["FP-002"]
}
```

必填字段：

| 字段 | 说明 |
|---|---|
| `stable_id` | 冻结项编号，格式为 `FP-001`，在同一结果内唯一 |
| `system` | 功能点所属系统，必须输出系统中文名，不得输出系统编码 |
| `level1_module` | Excel 一级模块 |
| `function_description` | 功能描述 |
| `count_item_name` | 功能点计数项名称 |
| `category` | 功能点类别 |
| `reuse` | 复用程度 |
| `change_type` | 修改类型 |
| `remark` | 类型、复用、修改类型和追溯依据 |
| `fact_ids` | 引用已存在的变更事实 |
| `route_ids` | 引用已存在的场景路由 |

可选或可为空字段：

- `level2_module`
- `level3_module`
- `level4_module`
- `system_scene_ids`
- `linked_process_ids`
- `linked_data_ids`

枚举值：

| 字段 | 允许值 |
|---|---|
| `category` | `ILF`、`EIF`、`EI`、`EO`、`EQ` |
| `reuse` | `高`、`中`、`低` |
| `change_type` | `新增`、`修改`、`删除` |
| `system` | 当前对外可选系统中文名，生产范围为 `车险理赔核心系统`、`非车险理赔核心系统` |

## 6. 数据功能追溯规则

数据功能必须能追溯到支撑过程或复核说明：

- `ILF` 默认关联维护它的 `EI` 或业务维护动作，在 `linked_process_ids` 中填写对应事务功能 `stable_id`。
- `EIF` 默认关联引用它的 `EI`、`EO` 或 `EQ`，在 `linked_process_ids` 中填写对应事务功能 `stable_id`。
- 不得为了配套数据功能硬补不存在的查询、提交、输出或维护过程。
- 支撑过程为既有未变更功能、无法相邻或资料未明确时，必须在 `remark` 中说明原因；若由 `review_notes` 说明，必须包含该条 `stable_id` 或 `count_item_name`，并写明资料不足、无法明确、列入复核或不硬补事务功能等原因。
- 事务功能如维护或引用数据功能，应在 `linked_data_ids` 中填写对应 `ILF` 或 `EIF` 的 `stable_id`。
- `linked_process_ids` 和 `linked_data_ids` 都必须引用同一组 `frozen_items` 中已存在的编号。

备注追溯要求：

- 数据功能备注应写明“关联过程：见 FP-xxx ...”。
- 事务功能备注如维护或引用数据功能，应写明“关联数据：维护/引用 FP-xxx ...”。
- 无明确支撑过程时，备注必须写明“支撑过程未在本次需求中明确，列入复核，不硬补事务功能”等同等含义。

## 7. 系统场景字典命名

当系统资料包含 `08-FPA场景拆分字典.md` 时：

- AI 必须优先匹配系统场景编号。
- 命中字典时，`system_scene_ids` 必须非空。
- `level1_module` 必须原样使用字典中的 `Excel一级模块`。
- `level2_module` 必须原样使用字典中的 `Excel二级模块`。
- `count_item_name` 必须原样使用字典中的 `功能点计数项名称`。
- 不得按需求章节、页面标题或临时理解重命名。

缺少系统字典时：

- 允许进入无系统字典模式。
- `AI评估.md` 必须说明资料缺口、临时归类依据和待复核点。
- 结构化结果仍必须满足事实、路由、冻结项和编号追溯规则。

## 8. assessment_context 与 project_features

`assessment_context` 至少应包含：

```json
{
  "system_code": "claimcar",
  "system_name": "车险理赔核心系统",
  "no_knowledge_mode": false,
  "target_person_days": 10
}
```

规则：

- `assessment_context.system_code` 必须等于当前任务系统编码。
- `assessment_context.system_name` 必须等于当前任务系统中文名。
- `frozen_items[].system` 必须等于当前任务系统中文名。
- `project_features` 只作为 AI 分析参考，不作为最终 Excel 参数来源。
- 页面选择的 `规模计数时机` 和 `完整性级别` 优先进入脚本 payload；未选择项使用模板默认值或平台兜底值。

## 9. 目标人天使用口径

`target_person_days` 是参考目标，不是硬性反推值。

AI 可以：

- 在事实支持范围内解释拆分粒度、复用程度、修改类型和复杂度。
- 在 `AI评估.md` 中说明目标人天与冻结清单结果的差异。
- 把目标偏离或资料不足写入 `review_notes`。

AI 不得：

- 为贴近目标新增、删除、拆碎、合并或重排功能点。
- 改变系统属性、复用程度或修改类型以凑值。
- 输出最终人天、目标命中、UFP、US、调整后功能点或 Excel 计算字段。

## 10. 提示词模板

固定提示词模板位于：

```text
data/modules/fpa/profile/skill/prompt_template.md
```

建议变量：

```text
{{ requirement_title }}
{{ input_text }}
{{ system_code }}
{{ system_name }}
{{ knowledge_summary }}
{{ system_scene_dictionary }}
{{ no_knowledge_mode }}
{{ target_person_days }}
{{ project_features }}
{{ schema_text }}
{{ output_rules }}
```

提示词必须明确：

- 只输出 `<AI评估.md>` 和 `<AI结构化结果.json>` 两个区块。
- `AI评估.md`、Excel payload 和过程 JSON 都只能来自同一组 `frozen_items`。
- 命中系统字典时必须原样使用字典中的 Excel 模块和计数项名称。
- 数据功能需要关联支撑过程或写明复核原因。
- 普通用户不需要看到中间排查产物。
- 提示词模板缺失时任务失败，不使用隐藏默认提示词。

## 11. AI评估.md 内容要求

`AI评估.md` 是后台评估说明和管理员排查依据，必须用简明中文说明：

1. 需求理解。
2. 资料使用情况，包括是否使用 `08-FPA场景拆分字典.md`。
3. 变更事实摘要。
4. 场景路由摘要。
5. 拆分/合并摘要。
6. 冻结功能点清单摘要，按 `ILF`、`EIF`、`EI`、`EO`、`EQ` 说明数量和含义。
7. 数据功能关联表，列出每个 `ILF` 或 `EIF` 的关联过程、缺失原因或复核提示。
8. 目标人天校准说明，明确目标只做参考，未改变冻结清单边界。
9. 待复核点。
10. 如果系统资料提供关键链路、后段链路或核心链路复核表，逐项说明是否涉及、是否单独计数、对应 `stable_id`、合并或不计原因。

不要单独生成候选功能点清单或审计说明；必要内容并入 `AI评估.md`。

## 12. JSON 校验

后端收到前端回传的 `AI结构化结果.json` 后，执行以下检查：

1. 是否为合法 JSON 对象。
2. 是否通过 `result.schema.json` 基础结构约束。
3. 顶层字段是否仅包含 v2 契约允许字段。
4. `schema_version` 是否为 `fpa.ai_contract.v2`。
5. `assessment_context` 是否绑定当前任务系统。
6. 事实、路由、决策和冻结项编号格式是否正确。
7. 所有引用编号是否存在且引用类型正确。
8. `frozen_items` 是否非空，必填字段和枚举值是否有效。
9. 命中系统字典的冻结项是否包含 `system_scene_ids`。
10. `ILF` 或 `EIF` 缺少 `linked_process_ids` 时，是否在备注中说明原因；若使用 `review_notes`，是否绑定该条 `stable_id` 或 `count_item_name` 并说明原因。
11. `linked_data_ids` 是否只引用已存在的数据功能。
12. 是否出现模板路径、人天、UFP、US、Excel 单元格等禁止字段。

失败时：

- 任务失败，不生成 Excel。
- 保存 AI 原始响应和校验错误。
- 普通用户看到脱敏后的错误原因和建议操作。
- 管理员可以查看完整校验错误和任务日志。

## 13. 与 Excel 生成链路的关系

Excel 生成只接受后端生成的脚本 payload。

边界如下：

- AI 输出 `AI结构化结果.json` 和 `AI评估.md`。
- 后端校验结构化结果，通过后从 `frozen_items` 构造脚本 payload 的 `items`。
- 后端补充任务配置、页面参数、模板路径、输出路径和 `target_work_days`。
- 脚本基于模板生成 `FPA工作量评估.xlsx` 和 `FPA生成过程.json`。
- 页面展示关键结果时，以脚本产物和后端汇总为准，不以 AI 原始输出为准。

禁止：

- 要求 AI 输出 Excel 公式或单元格位置。
- 要求 AI 计算人天或判断目标命中。
- 绕过后端校验直接把 AI 输出写入 Excel。
- 在脚本阶段新增业务功能点或判断拆分是否充分。

## 14. 与历史实现的差异

平台当前口径相对早期实现有以下变化：

- AI 顶层功能点明细改为 `frozen_items`，并由事实、路由、拆分/合并决策支撑。
- 旧的候选、覆盖和不确定性说明不再作为顶层契约；统一进入路由、拆分/合并决策、备注和 `review_notes`。
- `FPA生成过程.json` 由平台脚本生成，不由 AI 直接生成。
- 页面控制 `规模计数时机` 和 `完整性级别`，AI 的项目特征建议不作为最终参数来源。
- 平台不设置固定功能点条数上限，Excel 脚本通过动态增行支持超过模板默认明细行数。
- Excel 只能由脚本基于模板生成。
