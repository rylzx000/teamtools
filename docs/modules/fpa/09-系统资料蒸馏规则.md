# FPA 系统资料蒸馏规则

## 1. 文档目的

本文档定义 FPA 模块中“系统资料”如何从完整资料包压缩为 TeamTools 可用的精简系统知识包。

目标是：

- 后续每新增一个系统，都按统一规则生成系统知识包。
- FPA 生成时不把完整系统资料全量塞入 AI 请求包，避免提示词过长。
- 保留影响 FPA 判断的关键信息：系统边界、数据责任、业务过程、外部交互、复用口径。
- 让不同系统的资料结构、表达粒度和使用边界保持一致。

本文档不定义具体需求的 FPA 评估结果，不生成 Excel，不计算功能点和人天，不替代 `07-AI输出与提示词契约设计.md`。

## 2. 输入与输出定位

### 2.1 输入资料

系统资料蒸馏的输入优先来自 `fpa-skill` 已生成的完整系统资料包。

典型输入目录：

```text
某系统/
  系统资料/
    00-系统画像.md
    01-系统边界...
    02-数据功能...
    03-交易功能...
    04-外部系统...
    05-复用程度...
    ...
```

可参考的资料来源包括：

- 系统画像、系统边界、模块职责。
- 业务流程、典型业务链路。
- 数据对象、逻辑数据组候选。
- 交易功能候选，如 EI、EO、EQ。
- 外部系统、接口、批处理、报表、消息。
- 复用程度、修改类型、不计入功能点边界。
- 历史 FPA 审计说明和生成过程 JSON。

原始 PRD、设计文档、接口文档、表结构文档可以作为辅助资料，但平台首版不要求直接从原始资料全量生成系统知识包。优先流程是：

```text
原始系统资料
-> fpa-skill 完整系统资料包
-> TeamTools 精简系统知识包
-> FPA AI 请求包
```

### 2.2 输出资料

每个系统建议生成一个默认进入提示词的精简知识包：

```text
data/modules/fpa/knowledge/{system_code}/
  teamtools-system-brief.md
```

可选生成一个结构化索引，供后端或后续检索使用：

```text
data/modules/fpa/knowledge/{system_code}/
  teamtools-system-index.json
```

完整系统资料包或原始蒸馏来源统一放在：

```text
data/modules/fpa/knowledge/{system_code}/source/
```

MVP 阶段只强制要求 `teamtools-system-brief.md`。`teamtools-system-index.json` 可以后续根据检索和路由需要再补。

首批系统的精简知识包必须按以下目录命名：

| 系统编码 | 系统名称 | 精简知识包路径 |
|---|---|---|
| `claimcar` | 车险理赔核心系统 | `data/modules/fpa/knowledge/claimcar/teamtools-system-brief.md` |
| `claimoth` | 非车险理赔核心系统 | `data/modules/fpa/knowledge/claimoth/teamtools-system-brief.md` |
| `onlineclaim` | 在线理赔服务平台 | `data/modules/fpa/knowledge/onlineclaim/teamtools-system-brief.md` |
| `clqp` | 零配件报价系统 | `data/modules/fpa/knowledge/clqp/teamtools-system-brief.md` |

以上系统编码是配置、文件目录、接口入参和任务记录的统一口径，不得自行改名或新增别名。

后端读取精简知识包时，必须通过 `systems.yaml` 定位：

```text
TEAMTOOLS_DATA_DIR + knowledge_dir + brief_file
```

其中 `brief_file` 默认值为 `teamtools-system-brief.md`。`source/` 目录不默认进入 AI 请求包。

## 3. 精简知识包定位

`teamtools-system-brief.md` 是 FPA 专用系统知识简包。

它的用途是帮助 AI 在评估某个具体需求时理解系统背景，包括：

- 本系统负责什么。
- 哪些能力属于外部系统。
- 哪些数据由本系统维护，哪些数据只是引用。
- 常见业务过程、输入、输出、查询、状态变化。
- 哪些模块或链路通常可以复用。
- 哪些内容容易被误判成功能点。

它不是：

- 具体需求的 FPA 结果。
- 完整系统说明书。
- 全量接口清单。
- 表结构明细。
- 历史需求结论合集。
- Excel 生成契约。
- 人天估算规则。

## 4. 长度控制

系统知识简包必须控制长度，默认只保留会影响 FPA 判断的信息。

建议长度：

| 系统类型 | 建议长度 | 上限 |
|---|---:|---:|
| 专业工具或单一场景系统 | 3000-5000 中文字 | 6000 中文字 |
| 多流程支持系统 | 4000-8000 中文字 | 10000 中文字 |
| 复杂核心系统 | 6000-10000 中文字 | 12000 中文字 |

超过上限时，不继续堆正文，应拆为：

```text
teamtools-system-brief.md       # 默认进入提示词
teamtools-process-details.md    # 可选，流程细节
teamtools-interface-summary.md  # 可选，接口摘要
teamtools-data-objects.md       # 可选，数据对象摘要
```

FPA 请求包默认只放 `teamtools-system-brief.md`。只有当需求明确命中特定流程、接口或数据对象时，才考虑追加相关补充文件摘要。

如果 `teamtools-system-brief.md` 超过本节上限，不能靠后端静默截断处理，应先人工压缩或拆分补充文件，再重新生成精简知识包。

## 5. 标准结构

`teamtools-system-brief.md` 建议使用以下结构：

| 层级 | 标题 |
|---|---|
| H1 | `{系统名称} FPA系统知识简包` |
| H2 | `1. 系统定位` |
| H2 | `2. 系统边界` |
| H2 | `3. 核心模块` |
| H2 | `4. 关键业务链路` |
| H2 | `5. 数据责任摘要` |
| H2 | `6. 外部系统与接口摘要` |
| H2 | `7. FPA判断辅助` |
| H2 | `8. 复用程度判断口径` |
| H2 | `9. 常见误判提醒` |
| H2 | `10. 资料缺口与待确认点` |

如某系统资料很少，可以保留相同标题并写“资料未明确”。不要因为资料少而临时改变结构。

## 6. 各章节生成规则

### 6.1 系统定位

写清系统是什么、处在哪条业务链路、主要服务哪些用户或系统。

要求：

- 控制在 100-300 字。
- 不写宣传式总结。
- 不写技术栈细节，除非会影响 FPA 系统边界。
- 不输出人天、功能点数量、Excel 公式或模板信息。

示例口径：

```text
本系统主要承载车险理赔线上处理、任务流转、影像材料、查勘/定损协同等能力，是理赔主流程中的多流程支持系统。核心案件主数据通常由核心理赔系统维护，本系统侧重点是线上处理数据、任务状态、材料流转和外部交互。
```

### 6.2 系统边界

系统边界是知识包最重要的内容之一，必须明确：

- 本系统负责维护的业务能力。
- 本系统只查询、引用或展示的外部能力。
- 与核心系统、App、第三方服务、外部平台的责任边界。
- 边界不清的内容必须标为待确认。

生成时优先使用表格：

```markdown
| 边界对象 | 本系统责任 | 外部责任 | FPA影响 |
|---|---|---|---|
| 核心案件主数据 | 查询、展示、处理结果回写 | 核心理赔系统维护案件主数据 | 核心案件通常为 EIF；本系统处理结果写入通常为 EI/EO 候选 |
```

### 6.3 核心模块

只保留主要模块和职责，不展开所有子功能、字段、按钮、代码包。

建议字段：

```markdown
| 模块 | 主要职责 | 典型能力 | FPA关注点 |
|---|---|---|---|
```

模块职责要能帮助判断功能点，不要只是罗列模块名称。

### 6.4 关键业务链路

保留 3-6 条最能代表系统的数据流、状态流和业务过程的链路。

每条链路建议写：

- 触发方。
- 主要处理过程。
- 是否改变本系统数据或状态。
- 是否产生对外输出、通知、报表、清单、回写。
- 涉及的数据对象和外部系统。

不要把整条链路直接写成一个功能点。链路只是帮助后续识别 EI、EO、EQ、ILF、EIF 候选。

### 6.5 数据责任摘要

数据责任摘要用于辅助判断 `ILF/EIF`。

必须区分：

- 本系统维护的数据对象：可能是 `ILF` 候选。
- 外部系统维护、本系统引用的数据对象：可能是 `EIF` 候选。
- 只作为字段、参数、日志、中间表、缓存的数据：通常不单独计入。

建议表格：

```markdown
| 数据对象 | 维护责任 | 常见来源 | FPA候选 | 注意事项 |
|---|---|---|---|---|
```

要求：

- 不复制全量表结构。
- 不列字段明细，除非字段本身代表新的业务对象或边界。
- 不把外部接口返回字段直接写成 `EIF`。
- 不把日志表、缓存表、中间表、幂等表默认写成 `ILF`。

### 6.6 外部系统与接口摘要

接口摘要用于辅助判断 `EI/EO/EQ/EIF`。

只保留会影响 FPA 判断的交互：

- 外部系统调用本系统并改变数据或状态。
- 本系统调用外部系统获取主数据或业务结果。
- 本系统向外部系统输出处理结果、报文、通知、清单。
- 批量导入、批量导出、同步、回调、消息。

不要保留：

- 全量接口字段。
- HTTP 方法、完整 URL、参数类型细节。
- 和业务边界无关的技术接口。

建议表格：

```markdown
| 外部系统/调用方 | 交互目的 | 输入/输出摘要 | FPA影响 | 待确认点 |
|---|---|---|---|---|
```

### 6.7 FPA 判断辅助

本节输出候选背景，不输出具体需求的最终功能点结果。

固定分为：

```markdown
### 6.7.1 ILF 候选
### 6.7.2 EIF 候选
### 6.7.3 EI 候选
### 6.7.4 EO 候选
### 6.7.5 EQ 候选
```

每类只列常见候选和判断依据。

必须写明：

```text
以下内容只是系统级候选背景，不代表某个具体需求一定计入功能点。正式评估必须结合本次需求范围判断。
```

### 6.8 复用程度判断口径

本节用于辅助判断 `高/中/低` 复用程度。

建议按可复用资产整理：

```markdown
| 可复用对象 | 适用场景 | 倾向复用程度 | 降低复用的情况 |
|---|---|---|---|
```

要求：

- 复用口径必须基于已有模块、页面、接口、规则框架、数据对象或业务链路。
- 不要把“属于同一系统”直接等同于高复用。
- 对核心系统或复杂主链路，要提醒新增规则、状态、控制、审计、支付、理算能力可能降低复用。

### 6.9 常见误判提醒

本节专门压缩已有资料中的“不计入功能点边界”和经验规则。

必须覆盖：

- 单个字段通常不单独计功能点。
- 单个按钮、提示语、返回码、日志、异常处理通常不单独计功能点。
- 外部系统返回字段不直接构成 `EIF`。
- 查询若只是展示已有数据，通常为 `EQ`；若生成派生结果、报表、通知、文件，可能为 `EO`。
- 配置改造要看是否形成独立配置对象或配置维护过程。
- 历史 FPA 明细只能作为参考，不能作为系统永久规则。

### 6.10 资料缺口与待确认点

所有不确定内容必须集中列出。

建议格式：

```markdown
| 问题 | 影响 | 建议确认方式 |
|---|---|---|
```

待确认点不应在正文中被写成确定结论。

## 7. 证据与确定性规则

重要结论建议标注确定性。

确定性取值：

| 等级 | 含义 |
|---|---|
| 明确 | 资料直接写明 |
| 推断 | 多处资料支持，但没有直接定义 |
| 待确认 | 资料不足或存在冲突 |

建议写法：

```text
- 查勘任务由本系统维护，可能作为 ILF 候选。证据：明确，来源：系统资料/02-数据功能_ILF_EIF候选清单.md。
- 支付结果由外部支付系统返回，本系统主要引用展示。证据：推断，来源：接口边界资料。
```

MVP 阶段不强制每句话都标来源，但以下内容必须尽量标注依据或待确认：

- 系统边界。
- 数据维护责任。
- 外部系统责任。
- ILF/EIF 候选。
- 复用程度判断口径。
- 容易影响估算结果的业务链路。

## 8. 压缩规则

从完整资料包压缩为精简知识包时，按以下优先级保留信息：

1. 系统边界和外部系统责任。
2. 本系统维护的数据对象和外部引用数据对象。
3. 会改变数据或状态的业务过程。
4. 派生输出、通知、报表、清单、回写。
5. 查询、列表、详情、检索类能力。
6. 复用资产和降低复用的条件。
7. 常见误判和不计入边界。
8. 待确认问题。

以下内容默认删除或大幅压缩：

- 全量字段清单。
- 全量接口参数。
- 全量表结构。
- 长篇业务背景描述。
- 历史需求完整过程。
- Excel 模板说明。
- 人天计算、UFP/US 公式。
- 与 FPA 判断无关的技术实现细节。

## 9. 与 AI 请求包的关系

后端生成 FPA AI 请求包时，系统资料默认使用 `teamtools-system-brief.md`。

推荐组装方式：

```text
FPA 通用规则
+ 当前系统的 teamtools-system-brief.md
+ 本次用户上传/粘贴的需求资料
+ 页面配置项，如系统名称、规模计数时机、目标人天
+ 输出 JSON 契约
```

不得默认把完整 `系统资料/` 全量拼入请求包。

如本次需求命中特定流程、接口或数据对象，后续可扩展为：

```text
teamtools-system-brief.md
+ 命中的局部补充摘要
```

局部补充摘要可以来自人工维护的补充文件，也可以来自后端检索命中的资料片段。MVP 阶段先不强制实现检索。

异常处理规则：

- `knowledge_dir` 为空时，按无资料模式生成 AI 请求包。
- `knowledge_dir` 非空但 `teamtools-system-brief.md` 缺失时，视为系统资料配置错误，不按无资料模式继续。
- 精简知识包超过长度上限时，视为系统资料配置错误或待人工压缩事项，不自动截断。
- `source/` 下完整资料包只作为维护来源，不默认拼入 AI 请求包。

## 10. 新增系统流程

新增系统按以下流程接入：

1. 收集原始系统资料。
2. 使用 `fpa-skill` 或人工方式生成完整系统资料包。
3. 将完整系统资料包放入 `data/modules/fpa/knowledge/{system_code}/source/`。
4. 按本文档规则生成 `teamtools-system-brief.md`。
5. 人工快速复核系统边界、数据责任、外部系统、FPA 判断辅助。
6. 在 `systems.yaml` 中配置 `code`、`name`、`system_type`、`knowledge_dir`、`brief_file`。
7. 页面选择该系统后，后端将对应精简知识包放入 AI 请求包。

首版允许手动生成和维护 `teamtools-system-brief.md`。后续如系统资料频繁变化，再考虑做自动重建或局部更新。

## 11. 生成提示词模板

在新对话中生成某个系统的 `teamtools-system-brief.md` 时，建议使用以下提示词：

```text
请只生成系统精简知识包，不做具体需求 FPA 评估，不生成 Excel，不计算人天。

目标系统：
- 系统编码：{system_code}
- 系统名称：{system_name}

输入资料目录：
{完整系统资料包目录或 source 目录}

请按以下文档规则生成：
D:\project\teamtools\docs\modules\fpa\09-系统资料蒸馏规则.md

输出目标：
生成一个可保存为 teamtools-system-brief.md 的 Markdown 内容。

生成要求：
1. 只保留会影响 FPA 判断的系统背景。
2. 固定使用以下结构：
   - 系统定位
   - 系统边界
   - 核心模块
   - 关键业务链路
   - 数据责任摘要
   - 外部系统与接口摘要
   - FPA判断辅助
   - 复用程度判断口径
   - 常见误判提醒
   - 资料缺口与待确认点
3. 不输出具体需求的 FPA 结果。
4. 不输出 Excel 模板、公式、人天、UFP/US 计算。
5. 不复制全量接口字段、表结构明细或大段原文。
6. 重要结论尽量标明“明确、推断、待确认”。
7. 内容长度必须符合文档中的长度上限；超长时先压缩，不要继续堆正文。
8. 历史 FPA 结果只能作为经验参考，不得固化为系统永久规则。
```

## 12. 验收标准

一个系统知识简包可用，需要满足：

- 能在 5 分钟内让评估人员理解系统定位和边界。
- 能明确区分本系统维护数据和外部引用数据。
- 能提供常见 `ILF/EIF/EI/EO/EQ` 候选背景。
- 能说明哪些能力通常可以复用，哪些情况会降低复用。
- 能列出常见误判提醒，避免字段、按钮、日志、透传字段被机械计入。
- 能列出资料缺口和待确认点。
- 长度符合本文档限制。
- 不包含完整接口字段、表结构明细、Excel 公式、人天计算或具体需求 FPA 结果。

## 13. 维护要求

当完整系统资料包变化时，先判断变化是否影响 FPA 判断：

- 如果只改文字说明或非关键实现细节，可以不更新精简知识包。
- 如果改变系统边界、数据责任、外部系统、核心流程、复用口径，必须更新精简知识包。
- 如果新增重要模块、接口、数据对象或业务链路，必须更新精简知识包。
- 如果历史评估暴露出常见误判，应补充到“常见误判提醒”或“复用程度判断口径”。
