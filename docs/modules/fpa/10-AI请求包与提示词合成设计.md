# FPA AI 请求包与提示词合成设计

## 1. 设计目标

本文档定义 FPA 模块中 AI 请求包的生成边界、提示词合成规则、前后端职责、落盘文件和错误处理口径。

核心目标：

- 用户交互保持简单：上传或粘贴需求，选择系统，提交任务，最终下载 Excel。
- 后端负责生成 AI 请求包，前端不自行拼接完整提示词。
- 前端负责调用外部模型，并把模型响应回传后端。
- 后端负责校验 AI 结果，并通过脚本确定性生成 `FPA生成过程.json` 和 Excel。
- AI 只做业务功能点识别、参考目标校准和结构化判断，不生成 Excel、不输出最终人天、不决定公式。

## 2. 整体流程

MVP 流程如下：

```text
用户上传或粘贴需求
-> 用户选择系统和页面参数
-> 后端创建 FPA 任务
-> 后端读取系统配置、系统知识包、提示词模板和 JSON 契约
-> 后端生成 AI 请求包
-> 前端获取 AI 请求包并调用 DeepSeek
-> 前端回传 AI 原始响应或调用错误
-> 后端解析并校验 AI JSON
-> 后端生成 Excel 脚本输入 payload
-> 脚本生成 FPA生成过程.json 和 Excel
-> 用户下载 Excel
```

关键原则：

- AI 请求包是任务级快照，任务创建后不随资源文件修改自动变化。
- 已生成的 AI 请求包应保存到任务目录，便于管理员排查。
- 前端可展示、复制请求内容，但普通用户不通过文件列表下载 AI 请求包。
- 模型调用失败或 JSON 校验失败时，任务失败并保留排查信息。

## 3. 前后端职责

| 环节 | 前端 | 后端 |
|---|---|---|
| 收集需求 | 上传 Markdown 或粘贴文本 | 保存用户输入 |
| 选择系统 | 提供系统下拉选项 | 根据 `systems.yaml` 校验 `system_code` |
| 页面参数 | 提供可控参数，例如规模计数时机、目标人天 | 保存任务配置 |
| 读取系统资料 | 不直接读文件 | 读取 `teamtools-system-brief.md` |
| 合成提示词 | 不拼接完整提示词 | 读取模板并替换变量 |
| 调用模型 | 使用用户浏览器环境调用 DeepSeek | 不直接调用公网模型 |
| 回传结果 | 回传原始响应或调用错误 | 保存、解析、校验、生成结果 |
| 生成 Excel | 不生成 | 调用脚本确定性生成 |

前端可以提供“复制请求内容”的辅助能力，但复制内容必须来自后端生成的请求包，不允许前端临时拼接另一套提示词。

## 4. 输入来源

AI 请求包由后端基于以下输入合成：

| 输入 | 来源 | 是否必需 | 说明 |
|---|---|---:|---|
| 用户需求正文 | 用户上传或粘贴 | 是 | 需求原文，首版主要支持 Markdown 和纯文本 |
| 需求标题 | 用户输入或文件名 | 否 | 用于任务展示，AI 可辅助总结但不作为唯一来源 |
| `system_code` | 页面选择 | 是 | 首批固定为 `claimcar`、`claimoth`、`onlineclaim`、`clqp` |
| 系统知识包 | `systems.yaml` + `teamtools-system-brief.md` | 条件必需 | 有资料模式必需，无资料模式为空 |
| 页面参数 | 页面配置 | 是 | 例如 `count_timing`、`target_person_days`；目标人天作为参考目标进入模型正文，同时进入任务配置和脚本 payload |
| 提示词模板 | `data/modules/fpa/profile/skill/prompt_template.md` | 是 | 缺失则任务失败 |
| JSON 契约 | `data/modules/fpa/profile/schema/result.schema.json` | 是 | 用于约束 AI 输出 |
| FPA 规则摘要 | 资源包文档或模板内固定内容 | 是 | 供 AI 判断功能点类别、复用程度、修改类型 |

## 5. 系统资料读取规则

后端根据任务的 `system_code` 读取：

```text
data/config/modules/fpa/systems.yaml
```

再按以下规则拼接系统知识包路径：

```text
TEAMTOOLS_DATA_DIR + systems.yaml.knowledge_dir + systems.yaml.brief_file
```

规则：

- `brief_file` 未配置时默认使用 `teamtools-system-brief.md`。
- `knowledge_dir` 为空时进入无资料模式，`no_knowledge_mode = true`。
- `knowledge_dir` 非空但 `brief_file` 不存在时，任务失败，记录为系统资料配置错误。
- 系统知识包超过 `09-系统资料蒸馏规则.md` 定义的长度上限时，任务失败或记录管理员可见配置错误，不静默截断。
- MVP 默认只读取 `teamtools-system-brief.md`，不读取 `source/` 下完整资料。
- 不做片段检索，不把系统资料全文动态喂给模型。

首批系统编码固定如下：

| system_code | 系统名称 | 知识包目录 |
|---|---|---|
| `claimcar` | 车险理赔核心系统 | `modules/fpa/knowledge/claimcar` |
| `claimoth` | 非车险理赔核心系统 | `modules/fpa/knowledge/claimoth` |
| `onlineclaim` | 在线理赔服务平台 | `modules/fpa/knowledge/onlineclaim` |
| `clqp` | 零配件报价系统 | `modules/fpa/knowledge/clqp` |

代码、配置和文档不得自行引入 `claim_core`、`online_claim`、`other_system` 等未确认别名。

## 6. 提示词分层

AI 请求包逻辑上分为三层。

### 6.1 固定角色与边界

用于定义 AI 的角色和硬约束：

- 你是 FPA 功能点识别助手。
- 只输出符合契约的结构化 JSON。
- `items[].system` 必须输出系统中文名，例如 `在线理赔服务平台`，不得输出 `onlineclaim` 等系统编码。
- 不生成 Excel。
- 不输出最终人天计算结果。
- 不输出 UFP、US、总工作量或目标命中判断。
- 不输出模板路径、Excel 单元格、脚本参数或文件路径。
- 不确定内容应写入 `uncertainties`，不要强行判断。
- 未计数候选项应写入 `uncounted_items`，拆分覆盖和人工复核风险应写入 `quality_notes` 或 `coverage_notes`。
- 可以参考目标人天，在 FPA 框架内对功能点拆分粒度、复用程度和修改类型做可解释校准。
- 不得为了贴近目标人天无依据新增功能点、拆碎字段按钮、修改系统属性或编造业务范围。

### 6.2 任务上下文

用于描述本次任务：

- 需求标题。
- 用户需求原文。
- 已选系统名称和 `system_code`。
- 是否无资料模式。
- 页面可控参数。
- 目标人天作为“参考目标人天”写入模型正文，用于辅助 FPA 明细校准；目标命中判断仍由脚本完成。

### 6.3 判断资料与输出契约

用于提供判断依据：

- 已选系统的 `teamtools-system-brief.md`。
- FPA 分类和复用判断规则。
- 拆分完整性要求：阶段预览、最终预览、缺项提示、提交、配置查询、状态更新、允许缺项提交、补拍/继续提交等候选需要逐项判断。
- `result.schema.json` 对应的输出结构。
- 输出字段说明、枚举值、禁止项。
- JSON 返回要求。

如果模型接口支持 OpenAI 风格 messages，可以拆为 `system` 和 `user`；如果只支持单段文本，则由后端合成为一个完整 Markdown 文本。前端实际调用模型时只能选择 `messages` 或 `plain_prompt` 其中一种发送，不得把两种形态、`metadata` 或排查字段一并发送给模型。

## 7. 模板变量

提示词模板建议保留以下变量：

```text
{{ requirement_title }}
{{ input_text }}
{{ system_code }}
{{ system_name }}
{{ knowledge_summary }}
{{ no_knowledge_mode }}
{{ target_person_days }}
{{ project_features }}
{{ schema_text }}
{{ output_rules }}
```

变量说明：

| 变量 | 说明 |
|---|---|
| `requirement_title` | 任务标题，来自用户输入、文件名或后端生成 |
| `input_text` | 用户需求原文 |
| `system_code` | 页面选择的系统编码 |
| `system_name` | 系统显示名称 |
| `knowledge_summary` | 系统知识包正文；无资料模式为空或写明无资料 |
| `no_knowledge_mode` | 是否无资料模式 |
| `target_person_days` | 参考目标人天，可为空；供 AI 在 FPA 框架内做可解释校准，不要求 AI 输出 |
| `project_features` | 页面可控项目特征，例如规模计数时机 |
| `schema_text` | AI 输出 JSON 契约 |
| `output_rules` | 输出限制、枚举值和禁止项 |

模板变量缺失或替换失败时，任务失败，不使用隐藏默认提示词。`target_person_days` 是提示词正文变量，但必须以“参考目标”方式使用：先形成基于需求事实和系统资料的基准拆分，再在 FPA 框架内做可解释校准。

## 8. AI 请求包文件

每个任务目录下保存 AI 相关文件：

```text
data/tasks/fpa/{task_id}/ai/
  AI请求包.json
  AI请求摘要.json
  AI原始响应.json
  AI调用错误.json
  AI结构化结果.json
  AI分析.md
```

文件职责：

| 文件 | 生成方 | 说明 |
|---|---|---|
| `AI请求包.json` | 后端 | 前端调用模型所需的完整请求包 |
| `AI请求摘要.json` | 后端 | 脱敏摘要，便于排查请求构成 |
| `AI原始响应.json` | 前端回传、后端保存 | 模型原始返回 |
| `AI调用错误.json` | 前端回传、后端保存 | 浏览器调用模型失败时保存 |
| `AI结构化结果.json` | 后端 | 从模型响应中解析并通过 schema 校验后的标准 JSON |
| `AI分析.md` | 后端 | 可选，从 AI JSON 中提取或整理供页面查看 |

普通用户通过专用接口获取 `AI请求包.json` 的可调用内容，不在文件列表中展示该文件，也不提供下载入口。管理员可查看排查文件。

## 9. `AI请求包.json` 建议结构

MVP 建议结构：

```json
{
  "provider": "deepseek",
  "model": "deepseek-v4-flash",
  "request_format": "messages",
  "messages": [
    {
      "role": "system",
      "content": "固定角色和硬约束"
    },
    {
      "role": "user",
      "content": "任务上下文、系统资料、需求正文和输出契约"
    }
  ],
  "plain_prompt": "当模型入口不支持 messages 时使用的完整文本",
  "generation_config": {
    "temperature": 0.1,
    "top_p": 0.9
  },
  "metadata": {
    "task_id": "任务ID",
    "module": "fpa",
    "system_code": "claimcar",
    "system_name": "车险理赔核心系统",
    "no_knowledge_mode": false,
    "target_person_days": 10.0,
    "template_version": "文件时间戳或摘要",
    "knowledge_file": "modules/fpa/knowledge/claimcar/teamtools-system-brief.md"
  }
}
```

说明：

- `messages` 用于支持标准聊天接口。
- `plain_prompt` 用于网页端只接受单段文本的模型入口。
- 前端调用模型时必须按 `request_format` 二选一发送：`messages` 模式只发送 `messages`，`plain_prompt` 模式只发送 `plain_prompt`。
- `metadata`、`AI请求摘要.json`、任务路径、文件路径和排查字段不得发送给模型。
- `generation_config` 是建议值，实际前端调用时可以按 DeepSeek API 约束转换。
- `metadata` 只用于任务排查，不要求发送给模型。
- API Key 不写入请求包，不保存到后端。

## 10. 请求摘要

`AI请求摘要.json` 用于管理员排查，不保存完整需求正文和完整系统知识包。

建议结构：

```json
{
  "task_id": "任务ID",
  "system_code": "claimcar",
  "system_name": "车险理赔核心系统",
  "no_knowledge_mode": false,
  "input_chars": 12000,
  "knowledge_chars": 9800,
  "schema_chars": 3000,
  "prompt_chars": 26000,
  "template_file": "data/modules/fpa/profile/skill/prompt_template.md",
  "knowledge_file": "data/modules/fpa/knowledge/claimcar/teamtools-system-brief.md",
  "created_at": "2026-07-05T10:00:00+08:00"
}
```

摘要用途：

- 判断请求是否过长。
- 判断是否进入无资料模式。
- 判断使用了哪个系统知识包。
- 排查提示词模板和系统资料配置问题。

## 11. Token 与长度控制

MVP 不做复杂检索和自动压缩，但必须做长度保护。

建议控制项：

| 项目 | 处理规则 |
|---|---|
| 用户需求正文 | 完整写入请求包；如超过系统配置上限，任务失败并提示用户拆分 |
| 系统知识包 | 只读取 `teamtools-system-brief.md`；超出规则上限则任务失败 |
| JSON 契约 | 放入必要字段和枚举说明，不放无关实现细节 |
| 提示词模板 | 不写长篇解释，保留强约束和输出格式 |
| 总请求长度 | 后端记录 `prompt_chars`；超过配置阈值时任务失败 |

建议首版配置：

```yaml
max_input_chars: 30000
max_knowledge_chars: 12000
max_prompt_chars: 60000
```

说明：

- 上述是字符级保护，不等同于 token 精确计算。
- 字符数限制用于防止明显异常输入。
- 如果后续发现 DeepSeek 实际上下文更紧，应调整配置，不改代码。

## 12. 浏览器直连模型验收

由于首版由用户浏览器调用 DeepSeek，开发前必须验证目标模型入口是否允许浏览器直连。

验收项：

| 验收项 | 预期结果 |
|---|---|
| CORS | 浏览器页面可以直接调用模型 API，不被跨域策略拦截 |
| 鉴权 | API Key 可以通过浏览器请求头或模型入口要求的方式传入 |
| 响应格式 | 能拿到完整模型响应，且前端可以回传后端 |
| 错误信息 | 网络失败、鉴权失败、限流失败时能拿到可展示的脱敏错误摘要 |

当前已完成 DeepSeek 浏览器直连最小测试，确认前端到模型 API 的调用流程可用。后续只需在有效账号下补充验证成功响应结构。

如果 DeepSeek API 不允许浏览器跨域调用，首版主链路无法按当前方式实现，必须二选一：

- 使用允许浏览器调用的模型入口或企业代理入口。
- 改为后端模型代理模式，并重新评估服务器网络、API Key 保存和审计边界。

## 13. AI 输出要求

AI 必须返回可解析的 JSON，结构以 `07-AI输出与提示词契约设计.md` 和 `result.schema.json` 为准。

硬约束：

- 顶层必须包含 `items`。
- `items` 必须是非空数组。
- `items[].system` 必须是已知系统中文名，不允许使用系统编码。
- 每条明细必须包含功能点类别、复用程度、修改类型和判断依据。
- `category` 只能是 `ILF`、`EIF`、`EI`、`EO`、`EQ`。
- `reuse` 只能是 `高`、`中`、`低`。
- `change_type` 只能是 `新增`、`修改`、`删除`。
- `uncounted_items` 用于记录未计数候选项，包含 `description`、`reason`、`related_requirement_section`。
- `quality_notes` 用于记录人工复核提示，包含 `code`、`message`、`severity`，其中 `severity` 只能是 `low`、`medium`、`high`。
- `coverage_notes` 用于记录需求覆盖说明。
- 有目标人天时，AI 可以在功能点拆分粒度、复用程度、修改类型上做可解释校准。
- 校准必须基于需求事实、系统资料和 FPA 规则，不得为贴近目标无依据新增条目、降低复用或改变修改类型。
- 不输出汇总功能点。
- 不输出人天。
- 不输出 `target_work_days`、`target_hit`、`adjusted_work_days_middle`、`ufp`、`us`、`adjusted_fp`、Excel 路径、模板路径、单元格坐标或脚本参数。

如果模型额外输出 Markdown 说明，后端应优先提取 JSON；无法稳定提取时，任务失败并保存原始响应。

## 14. 错误处理

AI 请求包生成阶段错误：

| 场景 | 处理 |
|---|---|
| `system_code` 不存在 | 任务失败，提示系统配置不存在 |
| `knowledge_dir` 非空但知识包缺失 | 任务失败，提示系统资料配置错误 |
| 提示词模板缺失 | 任务失败，提示资源包配置错误 |
| JSON 契约缺失 | 任务失败，提示资源包配置错误 |
| 输入正文为空 | 任务失败，提示用户补充需求内容 |
| 输入或请求过长 | 任务失败，提示拆分需求或联系管理员调整配置 |

模型调用阶段错误：

| 场景 | 处理 |
|---|---|
| 前端调用 DeepSeek 失败 | 前端本地允许重试；用户放弃或达到重试上限后，前端回传 `AI调用错误.json`，后端标记任务失败 |
| 浏览器网络不可达 | 页面提示检查网络或 API Key |
| API Key 无效 | 页面展示脱敏错误摘要，不上传 API Key |
| 模型返回空内容 | 后端标记任务失败，管理员可查看原始响应 |

任务处于 `waiting_ai_call` 时，用户可以重新获取 AI 请求包并继续调用模型。用户关闭页面、离开页面或暂未配置 API Key 时，后端不自动判定失败；后续可通过任务保留期或管理员清理策略处理长期未回传任务。

结果解析阶段错误：

| 场景 | 处理 |
|---|---|
| 非法 JSON | 任务失败，可由用户重新调用模型 |
| JSON 外包裹说明文字 | 后端可尝试提取首个 JSON 对象；失败则任务失败 |
| schema 校验失败 | 任务失败，保存校验错误 |
| `items` 为空 | 任务失败，提示未识别到功能点 |
| 枚举值非法 | 任务失败，提示模型输出不符合契约 |
| `system` 输出系统编码 | 任务失败，提示必须输出系统中文名 |
| 出现 Excel 计算字段 | 任务失败，提示模型输出了禁止字段 |
| 复核字段结构非法 | 任务失败，提示 `uncounted_items` 或 `quality_notes` 不符合契约 |

## 15. 可见性与权限

普通用户：

- 可以查看任务状态。
- 可以查看脱敏错误摘要。
- 可以获取后端生成的 AI 请求内容用于前端调用。
- 可以查看 AI 分析摘要和最终生成结果。
- 可以下载 Excel。
- 不在文件列表中看到 `AI请求包.json`。
- 不下载排查文件。

管理员：

- 可以查看完整任务文件。
- 可以查看 AI 请求包、请求摘要、原始响应、调用错误和校验错误。
- 可以根据排查信息调整提示词模板、系统知识包或长度配置。

敏感信息规则：

- API Key 只在用户浏览器本地使用，不写入后端文件。
- 后端不得保存用户 API Key。
- 错误摘要不得包含完整 API Key、鉴权头、Cookie 或其他凭证。

## 16. 与 Excel 生成链路的关系

AI 请求包只服务于 AI 结构化识别，不直接服务于 Excel。

边界如下：

- AI 输出 `AI结构化结果.json`。
- 后端合并任务配置、页面参数、模板路径、输出路径和 AI 结构化结果，生成 Excel 脚本输入 payload。
- AI 的 `items` 应能无损映射到 Excel payload 的 `items`；`target_work_days`、`count_mode`、最终 `project_features`、模板路径和输出路径由后端提供。
- 脚本基于模板生成 Excel 和 `FPA生成过程.json`。
- 页面展示关键结果时，以脚本产物为准，不以 AI 原始输出为准。

禁止：

- 要求 AI 输出 Excel 公式。
- 要求 AI 输出 Excel 单元格位置。
- 要求 AI 计算人天。
- 要求 AI 判断是否命中目标人天。
- 直接把 AI 输出当成最终 Excel 明细，不经过后端校验和脚本标准化。

## 17. 后续扩展

后续可以扩展：

- 支持更多模型提供方。
- 支持模型接口配置化。
- 支持请求包预览和管理员调试模式。
- 支持更精细的系统资料片段检索。
- 支持多系统需求拆分。
- 支持提示词模板灰度和版本管理。

MVP 暂不做：

- 不做 Codex 或 Worker 后台执行链路。
- 不做 Redis 队列。
- 不做资料全文检索。
- 不做提示词版本库。
- 不做自动压缩和自动摘要。
- 不做多模型自动评测。
