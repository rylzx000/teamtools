# TeamTools 整体页面设计

## 1. 设计定位

TeamTools 是轻量内网工具平台，页面首版要先提供平台壳子，再承载各业务模块。

首版所有页面都应遵守同一套平台结构：

- 顶部栏。
- 左侧模块导航。
- 内容区。
- 公共错误页。

当前只有 FPA 一个业务模块，但页面结构要按后续可扩展多模块来设计，不能把整个平台写成单模块应用。

## 2. 页面清单

| 页面 | 建议路径 | 作用 |
|---|---|---|
| 登录页 | `/login` | 用户登录入口 |
| 平台首页或模块入口 | `/` 或 `/modules` | 进入平台后跳转到已启用模块 |
| 无权限页 | `/403` | 无权限访问提示 |
| 未找到页 | `/404` | 路由或资源不存在提示 |

平台首页首版可直接跳转到 `/fpa/tasks`，但不要把默认跳转写死成 FPA 专属逻辑。

## 3. 平台壳结构

### 3.1 顶部栏

顶部栏主要展示：

- 平台名称：`TeamTools`。
- 当前模块名。
- 当前用户展示名。
- 用户角色标识。
- 退出登录入口。

规则：

- 当前用户信息来自登录态。
- 管理员和普通用户共用同一套顶部栏。
- 退出登录后返回登录页。

### 3.2 左侧导航

左侧导航按模块组织。

规则：

- 只展示已启用模块。
- 当前模块和当前页面需要有明确选中态。
- 暂未开放模块可以不展示，或作为禁用占位，首版建议优先隐藏未启用模块。
- 普通用户和管理员共用导航，权限差异体现在数据范围和按钮可见性，不体现在导航结构本身。

### 3.3 内容区

内容区承载业务页面，建议统一支持以下页面模式：

- 列表页。
- 表单页。
- 详情页。
- 错误页。

通用约定：

- 使用面包屑或页面标题表达当前位置。
- 页面级操作按钮放在标题区右侧。
- 列表、表单和详情页都应保持简洁，不做营销型排版。

## 4. 通用交互规则

- 页面加载失败时显示轻量错误状态，而不是空白页。
- 列表页默认手动刷新，不自动轮询，除非模块页单独约定。
- 表单提交中需要显示忙碌态并禁用重复提交。
- 所有权限按钮都只做前端展示控制，最终权限以后端校验为准。
- 任何页面都不得显示模型 key、环境变量、服务器敏感路径或其他用户数据。

## 5. 通用空态和异常态

建议平台公共页面统一具备以下状态：

- 加载中。
- 无数据。
- 请求失败。
- 无权限。

这些状态尽量复用统一样式，避免每个模块自己发明一套提示样式。
