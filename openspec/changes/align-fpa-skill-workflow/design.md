## Context

TeamTools 当前 FPA MVP 已跑通“用户提交需求 -> 后端生成 AI 请求包 -> 浏览器调用模型 -> 后端接收结构化 JSON -> 脚本生成 Excel -> 用户下载”的链路。现有契约把 AI 输出重心放在 `items` 明细上，后端和 Excel 脚本围绕这些明细做校验、补备注、生成过程 JSON 和模板填充。

新版 `fpa-skill` 的核心变化是把业务判断前移并冻结：先形成变更事实清单，再做场景路由、拆分/合并判断和系统场景字典映射，最后得到冻结功能点清单。Excel 脚本只消费冻结清单，不再参与“拆得够不够、是否要补 ILF/EO、条数是否过少”等业务质量判断。

本 change 只建立文档和规格契约，不实现代码。后续实现时必须回源阅读新版 `fpa-skill` 的 `SKILL.md`、`references/fpa-assessment-rules.md`、`references/fpa-scenario-routing-table.md`、`references/excel-template-contract.md` 和 `references/audit-note-template.md`。

## Goals / Non-Goals

**Goals:**

- 明确 TeamTools FPA 的新目标流程：需求资料到变更事实、场景路由、冻结清单、`AI评估.md` 和 Excel。
- 明确 AI 一次响应中的 `AI评估.md` 内容和 `AI结构化结果.json` 内容边界。
- 明确系统资料精简包和 `08-FPA场景拆分字典.md` 需要支持系统场景字典和模块/计数项映射。
- 明确普通用户只看 `AI评估.md` 和 Excel，管理员可看后台排查 JSON。
- 明确页面首版开放的 Excel 参数：`规模计数时机` 和 `完整性级别`。
- 明确后续开发任务顺序，使提示词、schema、后端、脚本和测试能按同一契约实现。

**Non-Goals:**

- 不修改前端页面视觉稿和交互布局。
- 不实现在线维护系统资料、模板版本管理或 skill 历史版本管理。
- 不新增模型供应商、不改变浏览器直连模型的 API Key 处理策略。
- 不让 AI 生成 Excel、公式、模板路径、输出路径或最终计算字段。
- 不单独生成 `候选功能点清单.md` 或 `FPA审计说明.md` 作为普通用户产物；相关说明并入 `AI评估.md`。
- 不在本 change 中修改业务代码、脚本、模板或运行数据。

## Decisions

### 1. AI评估.md 作为用户可读说明，结构化 JSON 作为机器依据

后续实现中，AI 一次响应应包含两段内容：面向用户的 `AI评估.md` 正文，以及面向后端的 `AI结构化结果.json`。`AI评估.md` 用人能读懂的方式说明需求理解、系统资料使用情况、拆分过程、关键假设、目标校准、待复核点和功能点摘要；`AI结构化结果.json` 必须包含可追溯的变更事实、路由、拆分/合并决策和冻结功能点清单。

`候选功能点清单.md` 不再作为 TeamTools MVP 的独立产物。它在新版 `fpa-skill` 中承担的人类可读拆分说明能力，合并到 `AI评估.md`。Excel 明细、`AI评估.md`、后台过程 JSON 和脚本输入 payload 都必须来自同一组冻结清单，不允许在填表阶段临时增删、改名、重排或换类型。

每条冻结功能点必须建立字段级追溯关系：`stable_id` 唯一，`fact_ids` 和 `route_ids` 非空且引用有效；命中系统场景字典时还必须记录系统场景编号。缺少追溯关系、编号重复或引用悬空时，应在后端校验阶段失败，而不是进入 Excel 生成。

备选方案是继续让 AI 直接输出 `items`，再由脚本或后端补业务提示。该方案实现成本低，但会把业务拆分质量分散到提示词、校验脚本和 Excel 逻辑中，难以审计，也不符合新版 `fpa-skill`。

### 2. 系统资料提供“路由辅助”，不提供完整上下文全文

后端仍按用户选择的单一系统读取系统资料包。系统资料包的关键输入是 `08-FPA场景拆分字典.md`，用于稳定匹配系统场景编号、Excel 一级模块、Excel 二级模块和功能点计数项名称；`teamtools-system-brief.md` 用于提供系统边界、核心链路、后段链路、数据责任和外部接口等背景。

有系统资料时，提示词优先使用 `08-FPA场景拆分字典.md`，再结合系统简述。没有 `08-FPA场景拆分字典.md` 时允许继续生成，但必须标记为“无系统字典模式”，并在 `AI评估.md` 中说明临时归类依据和待复核点。

备选方案是动态读取完整系统资料或让前端上传完整资料给模型。该方案会导致提示词膨胀、成本上升和资料泄漏边界更难控制，不适合作为 MVP 默认方案。

### 3. Excel 脚本只做确定性生成和结构校验

脚本职责限定为：读取一次性 payload、复制模板、动态扩展/收缩明细区、写入项目特征和冻结明细、保护公式、生成过程 JSON、输出结构提醒。脚本不得根据核心系统、目标人天、关键词、类型分布或条目数量自动补功能点。

结构提醒可以覆盖备注完整性、目标命中、模板结构、公式保护和允许值错误。业务拆分质量必须在 AI 结构化结果和 `AI评估.md` 阶段通过事实、路由和拆分/合并决策解决。

首版页面只开放两个会直接影响 Excel 的项目特征下拉项，名称必须与模板一致：

- `规模计数时机`：默认 `1.21 估算中期`；可选 `1.39 估算早期`、`1.21 估算中期`、`1.10 估算晚期`、`1.00 项目交付后及运维阶段`。
- `完整性级别`：默认 `1.10 完整性级别为A/B同时为达成完整性级别要求采取了特殊的设计及实现方式`；可选 `1.00 没有明确的完整性级别或等级为C/D`、`1.10 完整性级别为A/B同时为达成完整性级别要求采取了特殊的设计及实现方式`、`1.30 完整性级别为A同时为达成完整性级别要求在软件开发全生命周期均采取了特定、明确的措施`。

其他 Excel 项目特征继续使用模板默认值。项目特征落 Excel 的优先级固定为：用户页面明确选择 > 模板默认值 > 平台兜底。AI 输出的 `project_features` 只作为分析参考和 `AI评估.md` 材料，不直接写入 Excel payload，也不得覆盖用户页面选择或模板默认值。

### 4. 用户可见产物和后台排查产物分层

普通用户只需要看到简化流程、结果摘要、`AI评估.md` 预览和正式 Excel 下载。页面不展示 `change_facts`、`routing_decisions`、`split_merge_decisions`、`frozen_items`、payload、完整过程 JSON 或路由结构。

后台建议保留 `AI结构化结果.json` 和 `FPA生成过程.json`，用于校验、复现和排查。AI 请求包、原始响应、脚本 payload、调用错误详情和日志属于排查产物，普通用户只看脱敏摘要，管理员可查看必要内容。这样保留定位问题的能力，也避免普通用户误用中间文件作为正式交付件。

Excel 脚本输入 payload 默认不保留为正式文件。它是一次性填表参数，应优先使用内存或运行时临时目录，脚本执行完成后删除；如为管理员排查临时保留，必须标记为 admin-only/internal debug/runtime temp，普通用户不可见、不可下载、不可作为下次评估输入。

`FPA生成过程.json` 可以作为脚本和后端之间的结构化过程产物，但普通用户页面不直接展示完整过程 JSON。需要页面摘要时，只返回脱敏结果摘要字段，不得暴露服务器绝对路径、payload 路径、内部 file role、原始模型错误、环境变量或其他排查细节。完整过程 JSON 或未脱敏字段仅管理员可见。

### 5. 目标人天进入提示词，但不得反向决定拆分

目标人天可作为 AI 校准参考，用于复用程度、修改类型、复杂度和规模计数时机的解释性调整；但不得为了贴近目标而新增、删除、拆碎或合并功能点，也不得虚构数据组、接口或业务输出。

## Risks / Trade-offs

- [Risk] AI 一次响应包含 Markdown 和 JSON，模型可能出现区块缺失或 JSON 不完整。→ Mitigation：提示词固定区块边界，后端分别提取 `AI评估.md` 和 `AI结构化结果.json`，schema 分层校验，失败时保留脱敏错误摘要和管理员排查文件。
- [Risk] 系统资料不包含场景字典时，路由质量可能下降。→ Mitigation：允许无字典模式，但必须记录资料缺口、假设和待复核项，不静默伪造系统映射。
- [Risk] `AI评估.md`、结构化 JSON 和 Excel 明细不一致。→ Mitigation：后端以结构化 JSON 的冻结清单生成 Excel，过程 JSON 保存 `stable_id` 和映射摘要，验证时检查数量、顺序、类型和备注一致。
- [Risk] 不保留脚本 payload 会降低排查效率。→ Mitigation：脚本 payload 不作为普通用户产物；后端可按管理员排查配置临时保存或登记为管理员可见内部文件。
- [Risk] 新契约会影响已有样例和测试。→ Mitigation：先更新 OpenSpec 和样例契约，再逐步改提示词、schema、后端、脚本和端到端验证。

## Migration Plan

1. 先完成本 OpenSpec change 的 proposal、design、delta spec 和 tasks。
2. 后续开发时先更新提示词与 JSON schema，使 AI 一次响应中的 `AI评估.md` 和 `AI结构化结果.json` 可独立提取和校验。
3. 再更新后端任务产物链路，使 `AI评估.md`、结构化结果、过程 JSON 和错误摘要按权限保存。
4. 再更新 Excel payload 和脚本，使其消费冻结清单并只做结构校验。
5. 最后补样例和端到端验证，确认生成 Excel 与模板保持一致。

回滚策略：若新链路验证失败，可保留当前 MVP 代码不变，仅不应用本 change 的实现任务；OpenSpec 仍记录目标契约，不应把未实现状态描述为已完成。

## Open Questions

- `AI请求包.json` 是否长期保留为管理员排查文件，需要在实现时结合排查体验和磁盘占用确认默认开关。
- 已有三个系统资料简包是否都已包含足够的系统场景字典，需要在后续实现任务中逐个检查。
